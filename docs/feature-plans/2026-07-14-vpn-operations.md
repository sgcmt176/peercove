# PeerCove VPN運用・安全性機能計画（Exit Node を除く）

- **作成日**: 2026-07-14
- **対象**: M3-12、M3-14e、M3-21〜M3-25
- **前提**: M3-20 までの `main`。実装開始時は最新の `roadmap.md`、
  `peercove-m0-handoff.md`、`decisions.md` を再読すること
- **方針**: 事業者サーバーを追加しない。Windows 10/11 と Ubuntu 22.04+
  を対象とし、macOS・モバイル・IPv6 は対象外

## 1. 目的

PeerCove は複数ネットワーク、直接通信、内蔵 DNS、サブネットルーター、ACL、
鍵ローテーション、チャット、ファイル転送まで実装済みである。次期トラックでは
通信機能を増やすだけでなく、次の 4 点を製品品質として強化する。

1. 接続できない理由を利用者自身が判断できる
2. 招待とバージョン差を安全に管理できる
3. 通信品質と公開サービスの状態を後から確認できる
4. 設定を安全に移行し、用途に応じて通信権限を細かく制御できる

## 2. 対象と対象外

### 対象

| 実装順 | ID | 機能 | 主目的 |
|---:|---|---|---|
| 1 | M3-12 | 更新通知・バージョン互換性表示 | 混在バージョンの早期発見 |
| 2 | M3-21 | 接続診断センター | 障害原因の特定と安全な診断出力 |
| 3 | M3-22 | 招待の期限・取消・参加承認 | 未使用・漏えい招待の無効化 |
| 4 | M3-23 | 品質履歴・ロス・ジッター | 不安定な通信の可視化 |
| 5 | M3-14e | DNS サービスヘルスチェック | 名前解決とサービス稼働の区別 |
| 6 | M3-24 | 暗号化バックアップ・端末移行 | 障害復旧と PC 交換 |
| 7 | M3-25 | ACL v2 | グループ・プロトコル・ポート制御 |

### 対象外

- Exit Node / `0.0.0.0/0` の広告
- 外部の常設コーディネーション、診断、監視サーバー
- コード署名前の自動更新適用
- macOS、モバイル、IPv6
- ブロードキャスト／マルチキャスト中継
- クラウドへの設定・チャット・統計の同期

## 3. 実装順と依存関係

```text
M3-12 バージョン情報
   └─ M3-21 診断センター
         ├─ M3-22 招待ライフサイクル
         └─ M3-23 品質履歴
               └─ M3-14e DNS ヘルスチェック

M3-24 バックアップ ── M3-22 の招待・鍵メタデータを含める
M3-25 ACL v2      ── M3-21 の診断と M3-23 の品質表示を利用
```

- M3-12 のバージョン情報を、M3-21 の互換性診断に再利用する。
- M3-23 は UI を閉じても履歴を取るため、UI ではなくデーモンで計測する。
- M3-14e は M3-23 の時系列保存・集約方式を再利用する。
- M3-25 は不達が ACL 由来か障害由来かを M3-21 で説明できる状態にしてから行う。
- M3-24 は M3-22 で増える招待状態を含む最終スキーマが固まってから行う。

## 4. 全機能に共通する設計原則

### 4.1 セキュリティとプライバシー

- 秘密鍵、PSK、招待トークンをログ、診断画面、通知、エクスポート用の平文
  manifest に出さない。
- 公開鍵、仮想 IP、外部 IP、ホスト名は診断上必要になりうるが、エクスポート時に
  「そのまま／マスク」を選べるようにする。
- 外部通信は更新確認だけを既定で許可する。DNS ヘルスチェックで外部 CNAME を
  検査する場合はレコード単位の明示設定を必要とする。
- 診断の既定動作は読み取り専用とする。管理者権限が必要な修復は、自動実行せず
  操作内容と影響を表示して個別確認を取る。
- 新しい外部 crate を使う場合は、実装時点のリリース状況、保守状況、既知の
  セキュリティ問題を確認して ADR に記録する。

### 4.2 互換性

- wire / IPC の追加フィールドは原則として `serde(default)` と
  `skip_serializing_if` を使い、旧版が無視できる形にする。
- 互換性を維持できない変更だけ `PROTO_VERSION` / `IPC_VERSION` を上げる。
- 新旧混在時は黙って機能を欠落させず、UI に「相手の更新が必要」と表示する。
- 設定ファイルを拡張するときは、旧版で読めなくなる影響を ADR と README に
  明記する。

### 4.3 UI/UX とアクセシビリティ

- 診断、品質、更新はネットワーク詳細のサイドバーにページとして配置し、
  モーダルの深い入れ子を避ける。
- 状態は色だけで示さず、アイコン、ラベル、説明文を併用する。
- アイコンは同一 SVG セットを使い、絵文字を操作アイコンとして追加しない。
- アイコンのみのボタンには `aria-label` とツールチップを付ける。
- 操作領域は 44 × 44 px を目安にし、キーボードの Tab 順序とフォーカスリングを
  検証する。
- 非同期ボタンは処理中に無効化し、近接位置に成功／失敗を表示する。
- グラフだけでなく最新値、平均、最大、欠測を文字でも表示する。
- ライト／ダーク双方で通常文字 4.5:1 以上のコントラストを確認する。
- `prefers-reduced-motion` を尊重し、状態更新でレイアウトが跳ねないようにする。

### 4.4 実装単位

- 1 ゴールを「設計 ADR → core → daemon/ops → IPC/DTO → UI → README →
  自動テスト → 実機検証」の順に進める。
- 将来の実装時はリポジトリ規約どおり 1 ゴール 1 コミットとする。本計画作成では
  コミットしない。
- OS API 境界、Windows デバイスループ、`WgBackend` の変更は
  `roadmap.md` の難易度別振り分けに従う。

## 5. M3-12 更新通知・バージョン互換性表示

### 5.1 ゴール

GitHub Releases から新しい安定版の存在を通知し、UI、デーモン、参加メンバーの
バージョン差と利用できない機能を明示する。自動ダウンロードと自動適用は行わない。

### 5.2 段階

#### M3-12a: ローカル更新確認

- Tauri 側で現在の UI バージョンとデーモンバージョンを取得する。
- GitHub Releases の `latest` 相当を HTTPS で確認する。
- 起動直後ではなく初回状態表示後に非同期実行し、3〜5 秒でタイムアウトする。
- 成功結果と最終確認時刻をローカルに保存し、24 時間以内は再取得しない。
- prerelease / draft は既定で無視し、SemVer で比較する。
- アプリ設定に「更新を確認する」を追加し、既定はオンとする。
- 更新ありの場合だけ、サイドバー下部と設定画面に通知する。
- リリースページを OS の既定ブラウザで開く。本文 HTML はアプリ内描画しない。

#### M3-12b: ネットワーク内バージョン表示

- `Hello` と台帳に、任意の `app_version` と `capabilities` を追加する。
- 旧メンバーは「バージョン不明」として通信を継続する。
- メンバー詳細にバージョンを表示し、機能不足だけ警告する。
- `capabilities` は文字列集合またはビット列のどちらかを ADR で選ぶ。
  旧版無視とデバッグ容易性から、初期案は安定した文字列 ID の集合とする。

### 5.3 主な変更箇所

- `crates/peercove-core/src/proto.rs`, `ipc.rs`
- `crates/peercove-cli/src/control.rs`, `daemon.rs`
- `apps/peercove-ui/src-tauri/src/lib.rs`, `dto.rs`
- `apps/peercove-ui/src/ipc.ts`, `prefs.ts`, `App.tsx`
- `apps/peercove-ui/src/components/PrefsDialog.tsx`, `TunnelView.tsx`

### 5.4 受け入れ条件

- オフラインや GitHub 障害で接続画面が遅延・失敗しない。
- 同じリリースを毎回通知しない。
- 旧デーモン／旧メンバーとの基本接続を維持する。
- バージョン文字列を信用して安全性判断をせず、表示と能力判定にだけ使う。
- 更新通知をオフにすると外部更新確認を行わない。

### 5.5 ADR

- **ADR-0029**: 更新情報取得先、キャッシュ、capability negotiation、
  コード署名前は通知だけに限定する理由

## 6. M3-21 接続診断センター

### 6.1 ゴール

「接続できない」「直接にならない」「DNSだけ動かない」を、ログを読み慣れていない
利用者が画面から切り分けられるようにする。

### 6.2 診断モデル

`peercove-core` に OS 非依存の表示モデルを置く。

```rust
DiagnosticReport {
    generated_at,
    scope,
    overall,
    checks: Vec<DiagnosticCheck>,
}

DiagnosticCheck {
    id,
    status: Pass | Warning | Fail | Unknown,
    summary,
    detail,
    action,
    evidence,
}
```

- `id` は UI 翻訳と将来互換のため固定文字列にする。
- `summary/detail` の生文字列を wire 正本にするか、コード＋引数を正本にするかを
  ADR で決める。多言語化を考慮し、推奨はコード＋構造化 evidence である。
- `Unknown` を失敗として扱わない。外部サーバーなしでは判定できない事項を
  推測で「成功」にしない。

### 6.3 診断項目

#### 共通

- UI からデーモン IPC へ接続できるか
- IPC / protocol / capability の互換性
- 設定ファイルの存在、読み取り、検証
- 秘密ファイルの存在と権限（内容は読んで表示しない）
- 実行時インターフェース、仮想 IP、MTU
- OS ルートの存在と競合候補
- Tailscale 等で既知の競合レンジがないか
- DNS リゾルバの待受、split DNS 設定、A/CNAME 解決
- 最終ハンドシェイク、RTT、直接／中継状態
- ACL による意図的遮断か

#### ホスト

- UDP 待受、UPnP 要求結果、リース状態
- 外部エンドポイントが観測されているか
- forwarding / relay の有効状態
- サブネットルーターの経路と重複

#### メンバー

- ホスト endpoint 候補と最後に成功した候補
- コントロールチャネルの接続状態
- 直接通信の試行停止理由、鮮度、クールダウン
- ホストから削除された状態

### 6.4 制約

- 常設の外部診断サーバーは作らないため、任意の UDP ポートがインターネットから
  到達可能かを単独端末だけで断定しない。
- 代わりに UPnP 成功、外部 endpoint 観測、実メンバーのハンドシェイクという
  「到達した証拠」を表示する。
- MTU は初期版では設定値・経路・既知問題を診断し、自動 Path MTU Discovery は
  別ゴールとする。

### 6.5 UI

- ネットワーク詳細に「診断」ページを追加する。
- 上部に総合状態、実行時刻、「再診断」「診断結果を保存」を置く。
- カテゴリは「アプリ」「トンネル」「インターネット」「DNS」「権限」に分ける。
- 問題だけを先に表示し、正常項目は折りたためるようにする。
- 修復操作は初期版では「設定を開く」「再接続」「ログを見る」だけに限定する。
- 診断出力の保存前に、含まれる情報とマスク状態を確認する。

### 6.6 診断レポート

- JSON の構造化レポートと、人が読める TXT を同梱する。
- 直近ログはレベルを含めて添付するが、既存の秘密情報非出力テストに加え、
  エクスポート前の deny-list / redact 処理をテストする。
- 設定は秘密フィールドを除いた要約だけを出す。
- UI から任意の保存先を選択する。自動アップロードはしない。

### 6.7 主な変更箇所

- `crates/peercove-core/src/ipc.rs` と新規 `diagnostics.rs`
- `crates/peercove-cli/src/daemon.rs`, `backend/`, `dns.rs`, `upnp.rs`
- `apps/peercove-ui/src-tauri/src/lib.rs`, `dto.rs`
- `apps/peercove-ui/src/components/DiagnosticsView.tsx`（新規）
- `apps/peercove-ui/src/App.tsx`, `ipc.ts`, `styles.css`, `i18n/ja.tsx`

### 6.8 受け入れ条件

- 正常、デーモン停止、設定不正、DNS不正、ホスト削除、ACL遮断をテストできる。
- 1 項目の診断失敗でレポート全体が失敗しない。
- 診断でトンネルや OS 設定を変更しない。
- エクスポートに秘密鍵、PSK、トークンが含まれないことを自動テストする。
- Windows / Ubuntu の実機で、同じ診断 ID が同じ意味を持つ。

### 6.9 ADR

- **ADR-0030**: 構造化診断モデル、OS 証拠の収集境界、秘密情報のマスク、
  外部診断サーバーを持たない制約

## 7. M3-22 招待の期限・取消・参加承認

### 7.1 ゴール

未使用招待の状態をホストが把握し、期限切れ・取消・初回利用済みを強制できるように
する。任意設定として、初回参加端末を承認まで隔離する。

### 7.2 段階

#### M3-22a: 招待台帳と期限

- トークン v3 にランダム `invite_id`、`issued_at`、`expires_at` を追加する。
- host.toml の peer に招待メタデータを保存する。秘密トークン本体は保存しない。
- 既定期限は 7 日。UI で 1 時間、1 日、7 日、30 日、無期限を選べるようにする。
- 期限切れ peer はデーモンがトンネルへ追加せず、ホスト UI に残して理由を表示する。
- 取消操作は peer と関連 PSK を既存 remove-peer と同じ安全な手順で削除する。
- 旧トークン v1/v2 は「期限なしの既存招待」として読み込めるようにする。

#### M3-22b: 一回利用と状態表示

- 最初に認証済み `Hello` を受けた時刻を `accepted_at` として保存する。
- 端末生成鍵へのローテーション完了後は、トークンに入っていた旧鍵での再接続を
  明示的に拒否する。既存ローテーションの挙動を回帰テストする。
- UI に「未使用」「接続待ち」「参加済み」「期限切れ」を表示する。
- 同じトークンを別端末で使った場合、先に参加した端末を優先し、後続を拒否する。

#### M3-22c: 任意の参加承認

- ネットワーク設定に「新しい端末を承認する」を追加する。
- 承認待ち peer はコントロールチャネルだけを利用できる隔離状態にする。
- Windows はユーザー空間デバイスで、Linux は firewall / route 層で、
  コントロール TCP 以外のトンネル通信を破棄する。
- 承認されるまで台帳、直接 endpoint、DNS、チャット、ファイル送信の対象にしない。
- ホスト UI に参加要求を表示し、「承認」「拒否」「詳細」を提供する。
- UI が閉じていても OS 通知を出せるよう、デーモン状態差分から通知する。

### 7.3 重要な設計判断

- 招待トークンには現在メンバー秘密鍵が入るため、期限はトークン内の自己申告だけで
  なくホスト側招待台帳で強制する。
- 「承認待ちだがホスト上の任意サービスへ到達できる」状態を許容しない。
  M3-22c は OS ごとの INPUT 相当の隔離が完成するまで完了扱いにしない。
- 時刻巻き戻りを考慮し、期限判定失敗時は安全側に倒す。ただし既に参加済みの端末を
  時計異常だけで突然削除しない。

### 7.4 主な変更箇所

- `crates/peercove-core/src/token.rs`, `config.rs`, `proto.rs`
- `crates/peercove-ops/src/invite.rs`, `peers.rs`
- `crates/peercove-cli/src/control.rs`, `commands/tunnel.rs`, `backend/`
- `apps/peercove-ui/src/components/InviteDialog.tsx`, `TunnelView.tsx`
- Tauri DTO / IPC / i18n

### 7.5 受け入れ条件

- 期限内、期限切れ、取消、二重利用、旧トークンを自動テストする。
- 期限切れ／取消済みトークンで台帳を取得できない。
- 承認待ちはホストの任意サービス、他メンバー、DNS、チャットへ到達できない。
- 承認・拒否が UI 再起動後も保持される。
- トークンをログや招待一覧へ再表示しない。

### 7.6 ADR

- **ADR-0031**: トークン v3、ホスト正本の招待状態、一回利用、承認待ち隔離

## 8. M3-23 品質履歴・パケットロス・ジッター

### 8.1 ゴール

現在の約 90 秒の UI 内スパークラインを、UI を閉じても収集されるローカル品質履歴へ
発展させ、ゲームやリモート操作の不調を説明できるようにする。

### 8.2 計測

- 既存 ping/pong に連番と送信時刻を持たせ、一定窓の RTT、欠落、順序逆転を集計する。
- 1 分窓で次を計算する。
  - RTT: latest / min / average / p95
  - jitter: 連続 RTT 差の平均（定義を ADR で固定）
  - packet loss: 送信 probe 数と期限内応答数
  - route: direct / relay / trying の滞在時間と切替回数
  - transfer: rx / tx の差分
- 制御チャネル切断とパケットロス 100% を区別する。
- プローブ頻度は既存制御接続への負荷を増やしすぎない値とし、初期案は 5 秒である。

### 8.3 保存

- デーモンがネットワーク別・ピア別にローカル保存する。
- 初期案は日単位 JSONL、保持 7 日、上限サイズを超えた古い日から削除する。
- 書き込みは一時ファイル＋rename または追記＋破損行無視で、異常終了に耐える。
- 公開鍵はローカル識別子として保持する。エクスポート時はマスク可能にする。
- チャット履歴や受信ファイルと保存先を分ける。

### 8.4 UI

- ネットワーク詳細に「品質」ページを追加する。
- 期間は直近 15 分、1 時間、24 時間、7 日を選べるようにする。
- RTT は折れ線、ロスは棒、経路切替は時系列イベントとして表示する。
- グラフ下に表形式の最新値、平均、p95、ロス、最終接続を表示する。
- 「ロス 5% 超が 3 窓継続」「直接から中継へ切替」などの通知は既定オフで、
  閾値と通知を設定できるようにする。

### 8.5 主な変更箇所

- `crates/peercove-core/src/proto.rs`, `ipc.rs`
- `crates/peercove-cli/src/control.rs`, `daemon.rs` と新規 `quality.rs`
- `apps/peercove-ui/src/components/QualityView.tsx`（新規）
- 既存 `history.ts` は短期描画キャッシュへ縮小するか、品質 API の結果へ置換する。

### 8.6 受け入れ条件

- UI を閉じた状態でもデーモン稼働中は履歴が記録される。
- デーモン再起動後も履歴が表示される。
- 欠測区間を 0 ms / 0% と誤表示しない。
- 直接／中継の切替時刻が診断ログと一致する。
- 7 日保持と容量上限が自動テストできる。

### 8.7 ADR

- **ADR-0032**: 品質指標の定義、probe 頻度、保存形式・保持期間、通知閾値

## 9. M3-14e DNS サービスヘルスチェック

### 9.1 ゴール

「DNS 名は引けるが、対象サービスは停止している」を区別し、登録済み URL の現在状態を
全メンバーへ表示する。

### 9.2 段階

#### M3-14e-a: 状態表示のみ

- `scheme` と `port` があるレコードを対象に、ホストが TCP connect を行う。
- 既定間隔 60 秒、timeout 3 秒、同時実行数に上限を設ける。
- 状態は `healthy / unhealthy / unknown / disabled` と最終確認時刻、応答時間を持つ。
- HTTP/HTTPS は初期版でも TCP 接続だけを既定とし、GET は送らない。
- メンバー参照がオフラインなら接続試験を行わず `unknown` とする。
- 状態を台帳の optional フィールドで配布する。

#### M3-14e-b: 任意の詳細チェック

- レコード単位で HTTP HEAD、期待ステータス、任意 path を設定できるようにする。
- 外部 CNAME へのチェックは既定オフとし、明示的に有効化した場合だけ外部へ接続する。
- 認証ヘッダー、Cookie、本文一致など秘密を持つ監視は対象外とする。

### 9.3 DNS 応答との関係

- 初期版では不健康でも A/CNAME 応答を取り下げない。
- 短時間の障害で DNS が揺れること、クライアントキャッシュ、全員からの到達性差を
  避けるためである。
- 将来「不健康時は回答しない」を追加する場合は別 ADR と実機検証を必要とする。

### 9.4 UI

- DNS カードに状態ラベル、応答時間、最終確認を表示する。
- ホストだけに「今すぐ確認」「ヘルスチェック設定」を表示する。
- URL コピーは状態に関係なく可能にし、不健康の場合は警告を添える。
- 色だけでなく「稼働中」「停止」「未確認」を文字で表示する。

### 9.5 主な変更箇所

- `crates/peercove-core/src/config.rs`, `dns.rs`, `proto.rs`, `ipc.rs`
- `crates/peercove-cli/src/dns.rs` と新規 `health.rs`
- `crates/peercove-ops/src/dns.rs`
- `apps/peercove-ui/src/components/DnsDialog.tsx`、Tauri DTO / IPC

### 9.6 受け入れ条件

- healthy、connection refused、timeout、オフライン、チェック無効をテストする。
- チェック失敗で DNS サーバーやトンネルが停止しない。
- 同時実行数と timeout が守られ、設定数に比例してタスクが無制限増加しない。
- 外部 CNAME へ既定で接続しない。
- 旧メンバーはヘルス情報を無視して従来どおり名前解決できる。

### 9.7 ADR

- **ADR-0033**: ヘルスチェック主体、間隔、外部接続、DNS 応答を自動変更しない方針

## 10. M3-24 暗号化バックアップ・端末移行

### 10.1 ゴール

ネットワーク設定と秘密ファイルを、利用者が指定したパスフレーズで暗号化して保存し、
別 PC または再インストール後に安全に復元できるようにする。

### 10.2 バックアップ形式

- versioned container とし、先頭に magic、format version、KDF parameters、salt、
  nonce を持つ。
- manifest と各ファイルは AEAD で暗号化し、改ざんを検出する。
- KDF は Argon2id、AEAD は XChaCha20-Poly1305 を第一候補とするが、crate 選定は
  実装時の保守状況確認後に ADR で確定する。
- パスフレーズ自体と復号鍵を保存しない。
- 平文一時ファイルを作らず、可能な限りストリームまたはメモリ上で処理する。

### 10.3 含めるもの

- 選択したネットワークの host/member TOML
- 参照される秘密鍵と PSK
- 招待メタデータ、DNS、ACL、グループ定義
- アプリ設定は任意

既定で含めないもの:

- 受信ファイル
- チャット履歴
- 品質履歴
- ログと診断レポート

### 10.4 復元

- 復元前にネットワーク名、役割、作成元 OS、含まれるカテゴリを表示する。
- 既存ネットワークへ黙って上書きしない。「別名で復元」「置換」「中止」を選ぶ。
- 復元した秘密ファイルへ Unix 600 / Windows のユーザー限定 ACL を設定する。
- メンバー設定を別端末へ移した場合は、接続後の鍵ローテーションを推奨し、
  ワンクリックで実行できるようにする。
- ホスト復元では peer、PSK、ACL、DNS の参照整合性を検証してから配置する。
- 復元途中の失敗は staged directory を削除し、既存設定を変更しない。

### 10.5 UI

- アプリ設定に「バックアップと復元」ページを追加する。
- バックアップ対象と、履歴を含めない既定値を明示する。
- パスフレーズ強度の断定表示は避け、長さと再入力一致を検証する。
- 復元前の確認画面と、完了後に必要な再接続／鍵更新を表示する。

### 10.6 主な変更箇所

- `crates/peercove-ops/src/backup.rs`（新規）を中心に純粋な列挙・検証を実装
- `crates/peercove-core` に format header / manifest の型を置くか ADR で判断
- `apps/peercove-ui/src-tauri/src/lib.rs`
- `apps/peercove-ui/src/components/BackupView.tsx`（新規）

### 10.7 受け入れ条件

- 正しいパスフレーズで Windows↔Ubuntu 間の復元ができる。
- 誤パスフレーズ、改ざん、途中切れを明確に拒否する。
- 秘密ファイルの権限が復元される。
- 既存設定を確認なしに上書きしない。
- テスト fixture に実在の秘密やユーザー環境のパスを含めない。

### 10.8 ADR

- **ADR-0034**: バックアップ形式、暗号方式、対象、クロス OS パス変換、原子的復元

## 11. M3-25 ACL v2

### 11.1 ゴール

現在のメンバー組ごとの全遮断を互換維持しつつ、グループ、方向、プロトコル、ポート、
サブネットを使ったルールをホストが管理できるようにする。

### 11.2 段階

#### M3-25a: グループと全通信ルール

- ホスト管理の member group を追加する。
- 既存 `deny = [(A, B)]` を読み込み、同等の v2 deny rule へ表示上変換する。
- source / destination に member、group、subnet を選べるようにする。
- 初期段階は protocol=`any` の allow / deny だけを扱う。
- 既存 ACL UI はルール一覧＋作成ウィザードへ置換する。

#### M3-25b: プロトコルとポート

- TCP、UDP、ICMP、any を扱う。
- TCP/UDP は単一ポートと範囲を扱う。
- ルール評価順、default action、双方向性を ADR で固定する。
- 推奨初期セマンティクス:
  - 上から最初に一致したルール
  - default allow（既存互換）
  - 方向あり（新規通信の開始方向）
  - 許可方向から開始したTCP/UDP/ICMPの応答はstatefulに許可

#### M3-25c: DNS サービス参照

- 宛先として DNS サービスレコードを選べるようにする。
- レコードの member/IP と port を保存時に検証し、名前変更へ追随する安定 ID を
  別途持たせる。
- CNAME 外部宛は PeerCove 内 ACL の対象外とする。

### 11.3 直接通信との整合性

ポート単位 ACL は WireGuard の `AllowedIPs` だけでは強制できず、直接通信はホストを
通らない。初期実装では、細粒度ルールが関係するメンバー組をホスト中継へ固定し、
ホストで全パケットを評価する。

- 台帳に `force_relay` と理由を配布する。
- UI は経路を「ACL により中継」と表示する。
- Windows ユーザー空間 relay と Linux firewall の両方で同じ evaluator の結果を
  使うか、共通 test vector で一致を保証する。
- 将来、全端末へポリシーを配布して endpoint 側でも強制できるようになった時だけ、
  細粒度 ACL と直接通信の併用を再検討する。

### 11.4 設定モデル案

```toml
[acl]
default = "allow"

[[acl.group]]
id = "servers"
members = ["<public-key>"]

[[acl.rule]]
id = "allow-game"
action = "allow"
source = { group = "players" }
destination = { group = "servers" }
protocol = "udp"
ports = ["25565"]

[[acl.rule]]
id = "deny-other-to-servers"
action = "deny"
source = "any"
destination = { group = "servers" }
protocol = "any"
```

最終スキーマは ADR で確定する。公開鍵を直接参照する箇所と、表示名／DNS 名を
参照する箇所を混同しない。

### 11.5 UI

- ルール一覧に「許可／拒否」「送信元」「宛先」「プロトコル」「ポート」「状態」を
  列表示する。
- 作成は段階式フォームにし、自然文プレビューを出す。
- ルール保存前に影響対象と「直接通信が中継へ変わる」ことを表示する。
- 並び替えはキーボードでも可能にする。ドラッグ操作だけに依存しない。
- メンバー行と診断画面から、遮断したルール ID を確認できるようにする。

### 11.6 主な変更箇所

- `crates/peercove-core/src/config.rs`, `proto.rs` と新規 `acl.rs`
- `crates/peercove-ops/src/acl.rs`
- `crates/peercove-cli/src/backend/windows/device.rs`, `backend/linux.rs`,
  `control.rs`, `direct.rs`
- `apps/peercove-ui/src/components/AclDialog.tsx` または新規 `AclView.tsx`
- Tauri DTO / IPC / i18n

### 11.7 受け入れ条件

- 旧 deny pair 設定が同じ遮断結果になる。
- 方向、TCP/UDP、単一ポート、範囲、ICMP、default action をテストする。
- Windows relay と Linux firewall に共通 test vector を適用する。
- 細粒度 ACL 対象の直接 peer が確立・残存しない。
- UI、チャット、ファイル送信、診断が同じ遮断理由を表示する。
- 設定再読込中に一時的な全許可状態を作らない。

### 11.8 ADR

- **ADR-0035**: ACL v2 スキーマ、評価順、既定動作、直接通信時の強制方法

## 13. 実機検証マトリクス

| 構成 | 必須確認 |
|---|---|
| Windows Host + Windows Member | 更新、診断、招待隔離、直接／中継、ACL |
| Windows Host + Ubuntu Member | クロス OS バックアップ、品質、DNS health、ACL |
| Ubuntu Host + Windows Member | Linux forwarding/firewall、診断、招待隔離 |
| Host + Member A + Member B | 招待二重利用、直接経路、ACL、品質切替 |
| オフライン環境 | 更新確認失敗が他機能へ影響しない |
| 時計ずれ | 招待期限が既参加端末を誤削除しない |

ネットワーク実疎通は自動テストだけで完了扱いにせず、各ゴールで README に再現手順を
追加してから実機検証を依頼する。
