# PeerCove M2(プロダクト化: デーモン分離 + デスクトップ UI)ハンドオフ資料

- **版数**: 1.0
- **作成日**: 2026-07-08(v0.1)/ 確定 2026-07-08(依頼者承認: 全項目推奨案)
- **状態**: **確定**
- **前提**: M0(技術検証)・M1(招待トークン/コントロールチャネル/台帳/削除/init)完了済み

---

## 1. 目的(3 行)

CLI を使えない人でも「アプリを起動 → 招待を貼る → 繋がる」を実現する。
そのために常駐デーモンと Tauri + React のデスクトップ UI に分離し、
M2 後半でインストーラ・自動起動まで含めて「配って使ってもらえる」形にする。

## 2. 確定済み方針(継承 + M2 で新規)

| 項目 | 方針 |
|---|---|
| 対応 OS | Windows 10/11、Ubuntu 22.04+(macOS は対象外のまま) |
| UI スタック | Tauri 2.x + React(方針まとめどおり) |
| UI 範囲 | **フル機能**: 接続/切断・メンバー一覧・招待(QR 表示)・参加・削除・設定編集・ログビュー・計測表示・トレイ常駐・自動起動 |
| 配布 | M2 後半でインストーラ(Windows: MSI / Ubuntu: deb)と自動起動まで。**コード署名は M3** |
| 商標 | 製品名・crate 名・バイナリ名に "WireGuard" を含めない(継続) |
| 秘密情報 | 秘密鍵・PSK・トークンを UI にも平文で常時表示しない(招待表示は明示操作 + ワンタイム的な扱い) |
| CLI | 廃止しない。デーモンの IPC クライアントとして再実装し、検証・上級者用に維持 |

## 3. アーキテクチャ方針

```
┌─ peercove-ui(Tauri + React、ユーザー権限)
│     │  ローカル IPC(JSON-RPC 風・行区切り JSON)
├─ peercove-cli(現 CLI の後継、ユーザー権限)──┤
│                                              ▼
└─ peercove-daemon(管理者/root 権限で常駐)
      ├─ 既存の WgBackend(Linux: kernel WG / Windows: wintun + boringtun)
      ├─ 既存のコントロールチャネル・台帳・supervisor
      └─ IPC サーバー(Windows: 名前付きパイプ / Linux: Unix ドメインソケット)
```

- **daemon は既存 `peercove-poc` のトンネル部分の移設**(supervisor・control・backend
  はほぼそのまま crate 分割)。将来の `peercove-daemon-win/linux` 分離前提を回収する
- IPC の認可: 同一マシンのユーザーのみ(パイプ/UDS のパーミッションで制御)。
  ネットワークへは公開しない
- UI は daemon が無いと動かない。daemon 未起動時は UI から起動を促す/起動する
  (権限昇格の扱いは §9 の未確定事項)
- status のファイル方式(5 秒周期)は IPC のイベント購読に置き換える(C-5 を吸収)

## 5. スコープ外(M2 ではやらない)

- コード署名・公証(M3)
- UDP ホールパンチング・直接通信(Phase 2 = トラック B)
- macOS / モバイル / IPv6
- 多言語化(まず日本語のみ。構造上妨げない程度の分離はする)
- アカウント/クラウド同期の類(サーバーレス方針の継続)

## 6. 受け入れ基準(M2 完了の定義)

1. CLI を一切使わずに、UI だけで「ホスト開始 → 招待 → 参加 → 疎通 → 削除」の
   E2E が Windows / Ubuntu 実機で再現できる
2. インストーラでインストール → OS 再起動 → 自動的に daemon が起動し、
   UI から前回のネットワークに再接続できる
3. 秘密鍵・PSK・トークンが UI・ログに不用意に表示されない(招待表示は明示操作のみ)
4. `peercove-cli`(旧 poc 相当)で M0/M1 の全操作が引き続き可能
5. アンインストールで daemon・サービス登録・トンネル残骸が残らない
6. IPC 設計・サービス化方式・インストーラ構成が ADR として記録されている

## 7. 既知のリスク・注意点

- **権限モデルが最大の設計点**: トンネル操作は管理者/root 必須。daemon を
  サービス化して UI は非特権で操作する形が本命だが、Windows サービスからの
  wintun 利用・セッション分離(Session 0)の挙動は早期に PoC すること
- Tauri 2.x の安定性・プラグイン(tray, autostart, notification)の成熟度を
  着手時に再調査(作業ルールの crate 調査と同様)
- wintun.dll の再配布ライセンス条件の確認(§9)。不可なら初回起動時ダウンロード方式
- 既存 CLI ユーザー(= 検証フロー)を壊さないこと。crate 再編後も
  README の検証手順が成立するよう随時更新する

## 8. M2 の順序と並行性

```
M2-G1(daemon 分離)── 土台。完了までは他ゴールに着手しない
   ├→ M2-G2(骨組み)→ G3 → G4 → G5 → G6(UI 系は直列でも並行でも可)
   └→ M2-G7 の前半(サービス化 PoC)は G1 直後から並行可
```

トラック B(直接通信)へ進む場合も M2-G1 の daemon 分離が前提になるため、
G1 は投資として最優先。

## 9. 確定事項(2026-07-08 依頼者承認)

| # | 決定 |
|---|---|
| Q1 | **OS サービス化**(Windows サービス / systemd)。UI は非特権のまま daemon を操作 |
| Q2 | **シンプル実用**(OS ネイティブ寄り)を基調。テーマ切替の余地は残すが M2 では作らない |
| Q3 | **1 つのアプリで両モード**(起動時/ホーム画面で「ホストする / 参加する」を選ぶ) |
| Q4 | **同時参加は 1 ネットワークまで**(daemon の状態 = Idle / Hosting / Joined の 3 値) |
| Q5 | **wintun.dll はインストーラに同梱可**。wintun.net の署名済みバイナリを無改変で side-by-side 配置し、Permitted API(wintun crate 経由)のみ使用するため Prebuilt Binaries License の範囲内。ライセンス文をアプリの謝辞に含める |
| Q6 | ノート PC 想定(1280×800 で全機能が収まるレイアウト) |
