## ADR-0001: WG バックエンド選定

- **日付**: 2026-07-07
- **状態**: 承認済み(2026-07-07 依頼者承認)

### 背景

M0 では Windows 10/11 と Ubuntu 22.04+ の双方で WireGuard プロトコルのトンネルを
CLI から作成・制御する必要がある。handoff 5.1 の第一候補は
「Linux: カーネル WG + netlink 制御 crate」「Windows: wintun + ユーザー空間実装(boringtun 系)」。
boringtun のメンテ状況は流動的とされていたため、着手前に最新状況を Web 調査した(2026-07-07 実施)。

### 調査結果(2026-07-07 時点)

| crate / プロジェクト | 最新版・日付 | ライセンス | 所見 |
|---|---|---|---|
| `boringtun` (Cloudflare) | 0.7.1 / 2026-05-01 | BSD-3 | 2024 年に放置疑惑(issue #407)→ その後再構築され 0.7 系でリリース再開。0.7 は `device` モジュールが削除され、プロトコルエンジン(`noise::Tunn`)+ `x25519` のみ。Windows 用デバイスループは元々無いので影響なし |
| NepTUN (NordSecurity) | v1.0.8 / 2025-07 | BSD-3 | boringtun フォーク。活発だが crates.io 未公開(git 依存が必要) |
| GotaTun (Mullvad) | 2025-12 発表 | — | boringtun フォーク。Android のみ・第三者監査前。汎用ライブラリとして未成熟 |
| `defguard_wireguard_rs` | 0.10.0 / 2026-06 | Apache-2.0 | 活発。Linux はカーネル WG を netlink で制御する高レベル API(インターフェース作成・ピア設定・統計取得)。Windows は wireguard-nt(wireguard.dll)方式で、`x86_64-pc-windows-gnu` ツールチェーン + MSYS2 が必要という開発摩擦あり |
| `wireguard-control` (innernet) | 2.0.0 / 2026-07-02 | LGPL-2.1+ | 活動あり。ただし LGPL で、innernet 向け設計。C ライブラリ依存の記述あり |
| `netlink-packet-wireguard` (rust-netlink) | — | MIT | 低レベル(netlink パケットの構築のみ)。rtnetlink と組み合わせるグルーコードが多い |
| `wintun` | 0.5.1 / 2025-01 | MIT | wintun.dll の安全なバインディング。安定してメンテされている |

### 選択肢

1. **Linux: `defguard_wireguard_rs`(kernel モード) / Windows: `wintun` + `boringtun` 0.7 の `noise::Tunn` + 自前デバイスループ**
2. 両 OS とも `defguard_wireguard_rs`(Windows は wireguard-nt 方式)
3. Linux: `netlink-packet-wireguard` + `rtnetlink` を直接使用 / Windows は 1 と同じ
4. Windows のエンジンに NepTUN(git 依存)を使用
5. Windows: wireguard-go サイドカー

### 決定(提案)

**選択肢 1** を採用する。

- Linux: `defguard_wireguard_rs`(カーネル WG を netlink 制御)
- Windows: `wintun` crate + `boringtun` 0.7 `noise::Tunn` + 自前の非同期デバイスループ
  (wintun 読み書き ↔ UDP ソケット ↔ Tunn、タイマー tick 含む)
- 両者は `peercove-poc` 内の `WgBackend` trait の背後に隠し、`#[cfg(target_os)]` で切替

### 理由

- Linux のカーネル WG は性能・安定性で最良、handoff の第一候補どおり。
  `defguard_wireguard_rs` は活発(直近 1 ヶ月にリリース)で、インターフェース作成〜
  ピア設定〜統計取得(`status` に必要)まで高レベル API で揃う。
  低レベル netlink crate 直叩き(選択肢 3)よりグルーコードが大幅に少ない
- Windows で `defguard_wireguard_rs`(選択肢 2)を使うと wireguard-nt 方式になり、
  windows-gnu ツールチェーン + MSYS2 必須という開発環境摩擦が大きい。
  また handoff は wintun + ユーザー空間実装を第一候補としている
- boringtun はリリースが再開(0.7.1 / 2026-05)しており、必要なのはプロトコル
  エンジン(`noise::Tunn`)のみ。デバイスループは Windows では元々自前実装が
  必要だったため、0.7 の `device` モジュール削除は影響しない
- NepTUN(選択肢 4)は crates.io 未公開で git 依存になるため第一候補にしないが、
  boringtun 0.7 の API に想定外の欠落があった場合の**フォールバック**とする
  (API は boringtun 系で互換性が高い)
- wireguard-go サイドカー(選択肢 5)は別バイナリの配布・プロセス管理が増えるため
  M0 では不採用

### 却下理由まとめ

- GotaTun: 未成熟(Android のみ・監査前・ライブラリとして未公開)
- `wireguard-control`: LGPL と innernet 向け設計。Apache/MIT 系で揃えられる代替がある

### 備考

- wintun.dll は再配布せず、開発機に手動配置(入手手順を README に記載)
- UPnP(G-6)の crate 選定は着手時に別 ADR として記録する(`igd` 系の最新状況を確認)

## ADR-0002: add-peer の反映方式と status の取得方式

- **日付**: 2026-07-07
- **状態**: 採用

### 背景

handoff 4.1 は `add-peer` について「実行中プロセスへ反映。M0 では設定ファイル追記+
再読込(SIGHUP or 再起動)でも可」とする。Windows バックエンドはユーザー空間実装の
ため、別プロセス(`add-peer` や `status` コマンド)から実行中トンネルの状態に直接
アクセスできない(Linux のカーネル WG は netlink で別プロセスからも操作できる)。

### 選択肢

**add-peer 反映**:
1. 設定ファイル追記のみ(ホスト再起動で反映)
2. 設定ファイル追記 + ホストプロセスが設定ファイルを定期再読込して新規ピアを動的追加
3. IPC(名前付きパイプ / UDS)でコントロールチャネルを作る

**status 取得**:
1. ホスト/メンバープロセスが定期的にステータスファイルへ書き出し、`status` はそれを読む
2. Linux はカーネルから直接読み、Windows のみステータスファイル
3. IPC

### 決定

- **add-peer**: 選択肢 2。`add-peer` は host.toml に `[[peer]]` ブロックを**追記**
  するだけ(root/管理者不要)。実行中の host プロセスは **5 秒間隔**で設定を再読込し、
  未登録の公開鍵のピアをバックエンドへ動的追加する。両 OS で同一の仕組み。
  ピアの**削除・変更**は反映しない(M1 のコントロールチャネルで実装)
- **status**: 選択肢 1 を基本とする。host/member プロセスが 5 秒間隔で
  `<config名>.status.txt` に書き出し、`status` コマンドはそれを表示する
  (ファイルの更新時刻が古い場合は警告)。正常終了時に削除する

### 理由

- IPC(選択肢 3)は M1 のコントロールチャネルと重複する先行投資になるため見送り
- 設定ファイル経由なら Windows/Linux で挙動が完全に一致し、検証手順も 1 つで済む
- 再読込間隔 5 秒は「メンバーの keepalive 25 秒」より十分短く、体感遅延にならない
- ステータスファイルは秘密情報を含まない(公開鍵・エンドポイント・転送量のみ)

### 却下理由

- 再起動方式(add-peer 1)はホスト経由の通信が数秒断になり、G-3 以降の検証で邪魔になる
- Linux だけカーネル直読み(status 2)は実装分岐が増える割に利点が薄い
  (カーネル直読みも root が必要な点は同じ)

## ADR-0003: ホストのフォワーディング(ハブ&スポーク)方式

- **日付**: 2026-07-08
- **状態**: 採用

### 背景

G-3 では Member A ↔ Member B が Host 経由で通信する(直接ピア設定なし)。
Host は「あるピアから受けたパケットを別のピアへ転送」する必要がある。
handoff 5.2-1 は OS の IP フォワーディング(Linux: sysctl、Windows:
`Set-NetIPInterface -Forwarding Enabled`)を想定し、自動設定する場合は
`down` での原状回復を求めている。

### 選択肢

1. 両 OS とも OS の IP フォワーディングを有効化(TUN に書き戻して OS に転送させる)
2. Windows はユーザー空間デバイス内で直接リレー / Linux はカーネル WG のため OS フォワーディング
3. 両 OS ともアプリ内リレー(Linux もユーザー空間実装に変える)

### 決定

**選択肢 2** を採用する。

- **Windows(ユーザー空間実装)**: 復号したパケットの宛先が**別ピアの AllowedIPs**
  に一致する場合、TUN(OS)へ渡さずデバイス内で直接暗号化して転送する。
  OS 設定を一切変更しないため、`down` での原状回復も不要
- **Linux(カーネル実装)**: host 起動時に
  `/proc/sys/net/ipv4/conf/<if名>/forwarding` へ `1` を書く(インターフェース単位)。
  この設定は**インターフェースの消滅とともに消える**ため、異常終了でも残骸が
  残らない。グローバルの `net.ipv4.ip_forward` は変更しない(他用途への影響と
  原状回復の複雑さを避ける)

### 理由

- Windows のトンネルは元々全パケットがユーザー空間を通るため、OS に出して
  戻すのは無駄が多い上に、`Set-NetIPInterface` の変更・weak host model・
  ファイアウォールの Forward 許可など環境依存の設定が増える。デバイス内リレーは
  コード ~30 行で、システム状態を汚さない
- Linux のインターフェース単位 forwarding は「incoming インターフェースの
  フラグ」だけで同一 IF 間のヘアピン転送が成立し、寿命がインターフェースと
  一致するため対のクリーンアップが自動で成り立つ

### 却下理由

- Windows の OS フォワーディング(選択肢 1): レジストリ/NetIPInterface の
  変更が残骸になりうる。異常終了時の原状回復が難しい
- Linux もアプリ内リレー(選択肢 3): カーネル WG の性能利点を捨てることになる

## ADR-0004: UPnP crate の選定と開放方式

- **日付**: 2026-07-08
- **状態**: 採用

### 背景

G-6 では Host 側で UPnP IGD によるポート自動開放を試行し、成否と外部エンドポイント
(推定)をレポートする。handoff は「igd 系 crate を使用。着手時に最新状況を確認」
としているため、2026-07-08 に Web 調査した。

### 調査結果(2026-07-08 時点)

| crate | 最新版・日付 | 所見 |
|---|---|---|
| `igd-next` | 0.17.1 / 2026-06-01 | 元 `igd` のコミュニティフォーク。累計 11.5M DL、libp2p-upnp 等が依存。同期/async 両 API |
| `igd`(sbstp) | 0.12.1 / 2023-05 | 3 年以上リリースなし。事実上停止 |
| `rupnp` | — | 汎用 UPnP。IGD 用途にはグルーコードが増える |
| `natpmp` 系 | — | NAT-PMP/PCP は別プロトコル(M0 対象外) |

### 決定

- **`igd-next`(同期 API)** を採用。ポート開放は host 起動時の一回処理なので
  async 統合は不要
- リース時間は **86400 秒(24h)**。`down`(Ctrl+C)で `remove_port` を呼んで
  対で削除し、異常終了で残ってもリース切れで自動消滅する
- 内部アドレスは「ゲートウェイへ向かうソケットのローカル IP」で自動判定する
- 失敗時のメッセージには次の行動(ルーターの UPnP 設定確認、手動ポート
  フォワード手順、CGNAT の可能性)を含める。外部 IP がプライベート/CGNAT
  レンジの場合は二重 NAT の警告を出す

### 却下理由

- `igd`: メンテ停止(最終リリース 2023-05)
- 無期限リース(lease 0): 異常終了時にルーターへ残骸が残る。24h リースなら
  最悪でも翌日に消える(一部ルーターは無期限しか受け付けないため、その場合のみ
  lease 0 へフォールバックする)

## ADR-0005: M1 の参加方式・招待トークン・コントロールチャネルの設計

- **日付**: 2026-07-08
- **状態**: 承認済み(2026-07-08 依頼者承認。参加方式=案 B、トークンはファイル既定 + --print/--qr)

### 背景

M1 では「招待トークン(pcv1)・トンネル内コントロールチャネル・台帳配布・
メンバー削除」を CLI で E2E 実装する(M0 handoff §8)。最大の設計分岐は
**メンバーの参加(join)をどう成立させるか**。WG はピアの公開鍵を事前登録する
必要があるため、「ホストがメンバーの公開鍵を知る手段」が必要になる。

### 選択肢(参加方式)

**案 A: 事前登録プロトコル方式**
- トークンには参加シークレット(乱数)とホスト情報を入れる
- メンバーは自分で鍵ペアを生成し、トンネル**外**の登録用ポート
  (例: UDP 51821)へ「公開鍵 + シークレットの MAC」を送る。ホストが検証して
  IP を割り当て、ピア登録して応答。その後メンバーがトンネル接続
- 利点: メンバーの秘密鍵がトークンに載らない。トークンをグループで使い回せる
- 欠点: **開放ポートが 2 つに増える**(UPnP/手動開放も 2 倍)。登録プロトコルの
  設計・実装・攻撃面(トンネル外で待ち受ける認証エンドポイント)が増える

**案 B: トークンにメンバー身元を同梱する方式(1 トークン = 1 メンバー)**
- `invite` 実行時にホストが**メンバーの鍵ペアと仮想 IP を先に生成・登録**し、
  トークンに「メンバー秘密鍵・割当 IP・ホスト公開鍵・エンドポイント一覧・
  (任意)PSK」をすべて同梱する
- メンバーは `join --token …` で設定ファイルと鍵を書き出すだけで接続できる
  (登録プロトコル不要、**追加ポートなし**)
- ピア登録は既存の「host.toml 追記 + 5 秒再読込」(ADR-0002)をそのまま再利用
- 利点: 実装が小さい、NAT 面が M0 から変わらない、失効 = remove-peer で自然
- 欠点: メンバー秘密鍵がトークンを経由する(≒ M0 の「設定手動コピペ」と同等の
  露出)。ホストがメンバーの秘密鍵を知る(ハブ&スポークでは全トラフィックが
  ホスト経由なので実害は限定的)。鍵ローテーションは将来コントロールチャネル
  上で実装可能

### 決定(提案)

- **参加方式は案 B** を採用(M1 の規模で確実に動くことを優先。鍵ローテーションを
  将来課題として明記)
- **pcv1 トークン形式**: 固定長中心の独自バイナリ(バージョン付き)を
  **base64url** で文字列化し `pcv1.` プレフィックスを付ける(約 160 文字)。
  受け渡し形態は依頼者要望により複数用意する:
  1. **文字列(base64url)** … コピペ・チャット貼り付け用
  2. **QR コード** … ターミナル表示(将来はモバイル読み取り用に PNG/SVG も)
  3. **ファイル** … `invite.token` として保存(パーミッション制限付き)
- トークンは秘密情報(CLAUDE.md の出力禁止対象)のため、**既定はファイル保存**とし、
  `--print`(文字列表示)/ `--qr`(ターミナル QR 表示)は明示オプトイン
- **台帳**: 独立ファイルは作らず **host.toml の `[[peer]]` を正本**とする
  (`name` フィールドを追加)。メンバーへ配布する台帳はここから導出
- **コントロールチャネル**: トンネル内で host(仮想 IP)が TCP 待受。
  トンネル内であること自体が WG による認証・暗号化済み。メッセージは
  JSON Lines(バージョンフィールド付き)。M1 の用途: hello(名前通知)、
  台帳スナップショット配布、削除通知
- **メンバー削除**: `remove-peer` が host.toml・実行中バックエンド
  (`WgBackend::remove_peer` を新設)・コントロールチャネル通知の 3 点を行う

### 却下理由(案 A)

- 開放ポート追加は UPnP 失敗環境での手動手順を倍にし、M0 で確認した
  「ホスト 1 台 1 ポートだけ開ける」というシンプルさを壊す
- トンネル外の認証エンドポイントはセキュリティレビュー対象が増える。
  案 B なら攻撃面は M0 と同一(WG ポートのみ)

## ADR-0006: 仮想 IP の既定レンジと初期化コマンド

- **日付**: 2026-07-08
- **状態**: 採用(下記メモの M1 対応)

### 背景

既定例 100.100.42.0/24 は Tailscale の CGNAT アンチスプーフィング
(100.64.0.0/10 を DROP)と衝突する(G-2 検証で発覚。下記メモ参照)。
また host.toml の手書きが招待フロー(invite/join)導入後も残っていた。

### 選択肢

1. 既定例を固定の別レンジ(例 10.100.42.0/24)へ変更
2. **ホスト初期化コマンド(`init`)がランダムなサブネットを生成**
3. 起動時にローカルインターフェースと衝突検査して自動選び直し

### 決定

- **選択肢 2**: 新コマンド `init` が host.key と host.toml を生成し、サブネットは
  **10.x.y.0/24(x = 64〜127、y = 0〜255 の乱数)** から選ぶ(ホスト = .1)。
  約 16,000 通りで、グループ間の衝突もほぼ起きない
- レンジの根拠: Tailscale の 100.64.0.0/10 外・家庭 LAN に多い
  192.168.0.0/16・10.0.x/10.1.x・Docker 既定の 172.17.0.0/16 をすべて回避。
  10.64〜127 は企業でも比較的使われにくい帯
- 加えて host / member の起動時、設定のサブネットが **100.64.0.0/10 内なら
  Tailscale 衝突の可能性を警告**する(既存設定の救済)
- 既存の 100.100.42.x の設定はそのまま使える(例とドキュメントのみ更新)。
  ローカルインターフェースとの自動衝突検査(選択肢 3)はインターフェース列挙
  crate が必要になるため M2 で検討

## ADR-0007: デーモン分離の移行戦略とローカル IPC の設計(M2-G1)

- **日付**: 2026-07-08
- **状態**: 採用

### 背景

M2 では UI(非特権)からトンネル(要管理者/root)を操作するため、常駐デーモンと
ローカル IPC が必要(M2 handoff §3、権限モデルは OS サービス化で確定)。
一方で既存 CLI の検証フローを壊さないことが受け入れ基準に含まれる。

### 決定

**移行は 2 段階(G1a/G1b)**:

- **G1a(今回)**: crate 構成は変えず、`peercove-poc daemon run` サブコマンドとして
  IPC サーバーを実装する。既存の `host`/`member`(プロセス内実行)はそのまま残し、
  検証フローを保つ。CLI に daemon 操作コマンド(status/start/stop)を追加
- **G1b(G7 のパッケージング時)**: `peercove-daemon` / `peercove-cli` に crate 分割し、
  サービス登録(Windows サービス / systemd)を実装。プロセス内実行モードは
  その時点で開発用フラグに格下げ

**IPC 設計**:

- トランスポート: Windows = 名前付きパイプ `\\.\pipe\peercove-daemon` /
  Linux = Unix ドメインソケット(既定 `/run/peercove.sock`、非 root 実行時は
  `$XDG_RUNTIME_DIR/peercove.sock`)
- フレーミング: **JSON Lines**(コントロールチャネルと同形式)。
  リクエストは `{"id":n,"req":{"method":…}}`、応答は `{"id":n,"ok":…}` か
  `{"id":n,"err":"…"}`
- メソッド(v1): `status` / `start_host {config, upnp}` / `start_member {config}` /
  `stop`(トンネル停止)/ `shutdown`(デーモン終了)。
  invite / join / remove-peer は**設定ファイル操作なので IPC 不要**
  (UI・CLI がユーザー権限で直接実行し、実行中デーモンは既存の 5 秒再読込で追随)
- デーモンの状態モデル: `Idle` / `Hosting` / `Joined` の 3 値(M2 handoff Q4)+
  直近の統計・台帳スナップショット
- 認可: ローカルユーザーのみ(パイプ/ソケットの既定 ACL)。単一ユーザー PC を
  前提とし、複数ユーザーの権限分離は将来課題として明記

### 理由

- 2 段階にすることで、最大リスクの「supervisor の非同期化・キャンセル可能化」を
  crate 移動と切り離して先にテストできる(diff も検証もしやすい)
- JSON Lines は M1 のコントロールチャネルと同じ形式で、実装・デバッグ手段を共有できる
- 設定ファイルを正本とする既存設計(ADR-0002)を IPC でも活かし、
  メソッド数を最小にする(招待・削除系を IPC に乗せない)

### 却下案

- 最初から crate 分割: 移動 + 機能追加が同時になり、検証フロー破壊のリスクが高い
- gRPC / MessagePack 等: 依存が重い。ローカル 1:1 の制御用途に JSON Lines で十分

### 追記(2026-07-08、M2-G2 着手時)

UI(別プロセス)から IPC クライアントを使う必要が出たため、**クライアント部分のみ
`crates/peercove-ipc` へ先行抽出**した(サーバーは引き続き `peercove-poc::daemon`)。
G1b の crate 分割を段階的に進める形で、既存の検証フローには影響しない。

### 追記(2026-07-08、権限モデルの実装)

昇格したデーモンの IPC 端点へ非特権クライアントが接続できるよう、明示的に
権限を与える必要がある(検証で発覚):

- **Windows**: 名前付きパイプに DACL `D:(A;;FA;;;SY)(A;;FA;;;BA)(A;;FRFW;;;AU)` を付与。
  **ACE に総称権(`GA`/`GR`/`GW`)を書くとオブジェクト固有権へマップされず拒否される**ため、
  必ず `FR`/`FW`/`FA` を使う(`FW` は `FILE_APPEND_DATA` = `FILE_CREATE_PIPE_INSTANCE` を含む)
- **Linux**: root が bind した UDS を `0666` にする

いずれも「同一マシンのローカルユーザーなら操作可」という単一ユーザー PC 前提。
複数ユーザーの権限分離は将来課題。

## ADR-0008: UI からの設定ファイル操作(M2-G3/G4)

- **日付**: 2026-07-08
- **状態**: 採用(依頼者承認: 設定の置き場所=両方 / トークンは発行直後のみ表示 / UI が init 相当を実行)

### 背景

M2 の受け入れ基準は「CLI を一切使わずに UI だけで E2E」。しかし init / invite /
join / remove-peer のロジックは binary crate(`peercove-poc`)に閉じており、
UI から呼べない。またこれらは ADR-0007 のとおり **IPC には乗せない**(設定ファイル
操作であり、実行中デーモンは 5 秒再読込で追随する)。

### 決定

1. **`crates/peercove-ops` を新設**し、設定ファイル操作(init / invite / join /
   add-peer / remove-peer / rename)を移設する。CLI と UI の双方が呼ぶ。
   関数は**表示を持たず構造体を返す**(CLI は整形して print、UI は DTO へ変換)
2. **設定の置き場所**: 既定はアプリのデータディレクトリ
   (Windows `%APPDATA%\…`、Linux `~/.config/peercove/`)。
   加えて UI から任意のファイルを選べる(「別の設定を使う」)。CLI は従来どおり `--config`
3. **招待トークンは発行直後のダイアログでのみ表示**(QR + コピー)。閉じたら再表示しない。
   トークンはメンバー秘密鍵を含む(ADR-0005)ため、画面・ファイルに残し続けない。
   UI はトークンをファイルへ保存しない(CLI の `--out` は維持)
4. **「ホストを始める」で設定が無ければ UI が init 相当を実行**(鍵生成 + ランダム
   サブネットの host.toml 作成)。ボタン 1 つで開始できる

### 理由

- ops crate 化は IPC クライアントの抽出(ADR-0007 追記)と同じ構図で、
  G1b の crate 分割を段階的に進めることになる
- トークンの寿命を短くするのはセキュリティ上の既定。再発行は `invite` を
  もう一度実行すればよく、失効は `remove-peer` で行える

### 却下案

- 設定操作を IPC に乗せる: デーモンを特権のまま設定ファイルの書き手にすると、
  権限昇格の面が広がる。UI はユーザー権限で自分の設定を書けばよい
- トークンをいつでも再表示: トークンファイルを保持し続ける必要があり、
  秘密情報の露出時間が延びる

## ADR-0009: ログの取り出し・RTT 計測・トレイ常駐(M2-G5/G6)

- **日付**: 2026-07-08
- **状態**: 採用

### 背景

G5(設定・ログ・計測)と G6(トレイ常駐・通知)の実装に、3 つの設計判断が要った。
いずれも「デーモン(特権・別プロセス)と UI(非特権)が分かれている」という
ADR-0007 の構造から来ている。

1. UI はデーモンの標準エラー出力を読めない。ログをどう届けるか
2. トンネルの往復時間(RTT)を、アプリ層の UDP プロキシを作らずにどう測るか
3. メンバーの参加・切断を誰が検知して誰が通知するか

### 決定

**1. ログはデーモン内のリングバッファ + IPC で取り出す**

`tracing` の Layer(`peercove-poc/src/logbuf.rs`)で直近 500 行をメモリに保持し、
`IpcRequest::Logs { after_seq }` で差分を返す(1 応答は最大 200 行)。
フィルタは標準エラー出力と共通(`--log-level warn` ならログビューも warn 以上)。
CLI にも `daemon logs [--follow]` を用意する。

**2. RTT はコントロールチャネルの ping/pong で測る**

`ControlMessage::Ping { nonce }` / `Pong { nonce }` を追加(双方向)。
host は接続中の各メンバーへ、member はホストへ 5 秒間隔で ping を打つ。
測定値は仮想 IP をキーに保持し、`PeerSummary.rtt_ms` として status に載せる。

**3. 参加・切断の通知は UI が status の差分から出す**

デーモンは状態を持つだけ。UI が 2 秒周期のポーリングで台帳の `online` の
変化を見て OS 通知を出す。UI はウィンドウを閉じてもトレイに常駐して動き続ける。
通知の送出は Rust 側の `notify` コマンドで行い、frontend は
`@tauri-apps/plugin-notification`(npm)に依存しない。

### 理由

- **ログをファイルに書かない**: root/管理者が作ったファイルを非特権 UI が読む
  権限問題と、ローテーション・削除漏れ(「残骸を残さない」方針)を新たに抱える。
  常駐前提なのでメモリ保持で困らない。秘密鍵・PSK・トークンはそもそもログへ
  出さない方針(CLAUDE.md)なので、溜まった行はそのまま UI へ渡してよい
- **RTT に ICMP を使わない**: raw socket は特権が要り、非特権 UI から測れない。
  コントロールチャネルは既にトンネル内 TCP で常時張られており、WG により
  暗号化・認証済み。ping/pong は 5 秒に 1 往復・数十バイトで無視できる負荷。
  「アプリ層 UDP プロキシは作らない」方針(CLAUDE.md)とも整合する
- **`Ping`/`Pong` で `PROTO_VERSION` を上げない**: 追加メッセージなので、
  知らない旧実装は解析に失敗して黙って無視する(`Err(e) => debug!` の枝)。
  新ホスト × 旧メンバーでは RTT が `None` になるだけで通信は壊れない
- **通知をデーモンから出さない**: G7 で Windows サービス化すると Session 0 で
  動くため、デスクトップへ通知を出せない。検知も通知も UI 側に置けば、
  サービス化しても壊れない
- **ウィンドウを閉じても終了しない**: トンネルはデーモンが維持するので UI を
  閉じても切断されないが、UI が生きていないと参加・切断の通知が出せない
- **通知を Rust 側から出す**: frontend が npm の通知プラグインを import すると、
  `npm install` を忘れた環境で UI 全体がビルドエラーで起動しなくなる
  (Linux 検証で実際に踏んだ)。Rust の `tauri-plugin-notification` は Cargo が
  自動で解決するので、依存の取りこぼしが起きない。デスクトップでは許可の
  問い合わせも不要なので、JS 側の API を使う利点が無い

### 却下案

- **ログをファイル + tail**: 上記の権限とクリーンアップの問題。将来サポート用に
  ファイル出力が要るなら、デーモン側のオプションとして足す
- **ログを IPC のイベント購読(push)にする**: リクエスト/レスポンスだけの今の
  IPC に双方向ストリームを導入する必要がある。`after_seq` の差分取得なら
  取りこぼしも検知できる(`dropped`)ので pull で十分
- **RTT を台帳(`LedgerEntry`)に載せる**: 5 秒ごとに値が変わるため
  `send_if_modified` が毎回発火し、台帳が全メンバーへ再配布されてしまう。
  RTT は IPC の `PeerSummary`(デーモン → 自分の UI)にだけ載せる
- **WG のハンドシェイク間隔から RTT を推定**: 精度が出ず、keepalive 間隔
  (25 秒)に律速される

### 実装上の落とし穴(RTT 追加時に踏んだ 2 件)

コントロールチャネルに定期 ping を足したことで、それまで表面化していなかった
2 つの罠が顕在化した。**制御接続に定期メッセージを足すときは必ず確認すること。**

1. **`AsyncReadExt::take(n)` の上限は reader の「累計」**であり、1 行ごとではない。
   台帳が稀にしか流れなかった間は無害だったが、5 秒ごとの pong を足すと
   64 KiB を数時間で使い切り、`read_line` が 0 を返して(= EOF と誤認して)
   制御接続が落ちるようになる。**1 行読むごとに `set_limit` で戻す**。
   改行なしで返ってきたときは `limit() == 0` で「長すぎる行」と「途中で切断」を
   区別する
2. **`read_line` はキャンセル安全でない**。`select!` の分岐に置くと、他の分岐が
   先に完了したときに読みかけのバイトが失われる。ping のタイマー分岐を足すと
   5 秒ごとにその機会が生まれる。**読み側は専用タスクの素直なループで回し、
   `select!` にはキャンセル安全な `watch::changed` / `mpsc::recv` /
   `interval.tick` だけを置く**

読み書きをタスク分割した結果、両者が終了を競うようになったため、
`select!` に `biased;` を付けて「意味のある終了」を出す側(host = 書き、
member = 読み。どちらも `Removed` を扱う側)を先に見るようにしている。
読み側が終わると `Outbox` が落ちて書き側も即 ready になり、
`biased` が無いと終了理由が非決定になる。

### 影響

- Linux でトレイを使うには、ビルド時に `libayatana-appindicator3-dev` が要る
  (`tauri` の `tray-icon` feature)。GNOME では AppIndicator 拡張も要る
  (Ubuntu は既定で導入済み)
- 設定編集(`peercove-ops::settings`)で変えられるのは自分側の設定だけ。
  MTU・待受ポート・ホストのエンドポイントはインターフェース生成時に決まるので
  **再接続まで反映されない**。UI は保存時にその旨を出す

### 実機検証フィードバックで直した 3 件(2026-07-09)

Linux 実機で G5/G6 を動かしての指摘。いずれも「デーモンとポーリング型 UI」の
組み合わせ特有の穴だった。

1. **メンバー削除がメンバー側に伝わらない**。`Removed` を受けてもトンネルは
   張ったまま(通信だけ死ぬ)で、UI は「参加中」表示のままだった。
   → 制御チャネルの受信で `removed` フラグを立て、snapshot →
   `TunnelInfo.removed` → UI DTO と流し、UI が赤いバナー「ホストから削除
   されました」+ 切断ボタンを出す。削除後は再接続もやめる(ホストが WG ピアを
   消すので成功しないうえ、無駄なリトライを続けないため)

2. **設定ダイアログの入力欄が約 2 秒ごとにフォーカスを失う**。`Modal` の
   フォーカス当ての `useEffect` の依存に `onClose` を入れていたため、親(App)が
   2 秒周期の status ポーリングで再レンダーするたびに `onClose` の識別子が変わり、
   effect が再実行されて `dialog.focus()` が入力欄からフォーカスを奪っていた。
   → フォーカスはマウント時 1 回だけ(`[]` 依存)。キー購読の effect と分離する。
   **ポーリングで再レンダーする画面にモーダルを重ねるときの一般的な落とし穴**

3. **通知の穴**:
   - **削除で「切断」通知が出ない**: `diffMembers` は台帳に残るメンバーの
     online→offline しか見ておらず、台帳から消えた(=削除)メンバーを拾って
     いなかった。→ 「前回いて今回いない」も「切断」として拾う
   - **自分の出入りで鳴る**: メンバー自身も台帳に載るため。→ `selfAddress`
     (自分の仮想 IP)で自分をスキップ

### 通知の既知の制約(仕様として残すもの)

- **メンバーが自分から切断した場合の「切断」通知は最大 180 秒遅れる**。WG は
  明示的な切断信号を持たず、ホストは最終ハンドシェイクが `ONLINE_THRESHOLD`
  (180 秒 = WG セッション有効期限)を過ぎて初めてオフライン判定する。即座に
  反映されるのは「ホストによる削除」(台帳から消える)だけ。180 秒以内の
  切断→再接続はオンラインのままなので通知されない。**速くしたければ、既に
  ある ping/pong に read タイムアウトを足してハートビート死活判定にするのが筋**
  (online 判定を制御チャネル在線に寄せる。今回は online 判定の意味を変える
  リスクを避けて見送り)
- **Windows の dev では通知元が「PowerShell」になる**。`tauri-plugin-notification`
  は exe が `target/debug` / `target/release` から起動されているときは
  AppUserModelID を設定しない(`notify-rust` が既定の PowerShell AUMID を使う)。
  **G7 のインストーラがショートカットで AUMID を登録すれば「PeerCove」になる**。
  Linux は最初から製品名で出る

## ADR-0010: 配布形態とデーモンのサービス化(M2-G7)

- **日付**: 2026-07-09
- **状態**: 採用(依頼者承認: インストーラ主配布 + ZIP は上級者向け)

### 背景

M2 の受け入れ基準は「インストール → OS 再起動 → デーモンが自動起動 → UI から
再接続」と「アンインストールで残骸ゼロ」。ZIP 直配布ではサービス登録(= 管理者
権限でのデーモン常駐)・通知の AppUserModelID・クリーンアンインストールが
成立しないため、インストーラを主配布とする。

### 決定

1. **配布**: 主 = インストーラ(Windows MSI / Ubuntu deb)。副 = ZIP
   (上級者・検証用。`daemon service-install` か手動 `daemon run` で使う)
2. **Windows サービス**: `windows-service` crate 0.8(Mullvad 製。同社 VPN が
   「wintun を使うデーモンのサービス化」という同一用途で本番使用)。
   - アカウントは **LocalSystem**(wintun のアダプタ作成・ルーティング操作に必要)
   - `daemon service`(hidden)が SCM エントリ。停止・シャットダウン制御で
     `daemon::serve()` の停止シグナルを送り、トンネルのクリーンアップを走らせて
     から STOPPED を報告する(STOP_PENDING の wait hint 30 秒)
   - 既存の名前付きパイプ + SDDL(ADR-0007)がそのまま「SYSTEM のデーモン ↔
     非特権 UI」で機能する
3. **秘密鍵の ACL に SYSTEM(SID `S-1-5-18`)を追加**(`peercove-ops::secret`)。
   従来は「現在のユーザーのみ」で、LocalSystem のサービスが鍵を読めなかった。
   SID 表記なのは「SYSTEM」という名前が OS 言語でローカライズされるため。
   **この変更より前に作られた鍵ファイルには手動で付与が必要**(README 参照)
4. **Linux**: systemd ユニット(`packaging/systemd/peercove-daemon.service` が
   単一ソース)。deb はそのまま同梱、CLI の `daemon service-install` は
   ExecStart を実行中バイナリのパスに置換して `/etc/systemd/system/` へ設置
5. **`daemon service-install` / `service-uninstall`** を CLI に追加(両 OS)。
   ZIP 配布と開発時 PoC 用。MSI / deb はそれぞれの機構(WiX ServiceInstall /
   unit 同梱 + maintainer script)で登録し、**サービス名(`peercove-daemon`)と
   起動引数は CLI と同一**にする(`service.rs` の定数が正)
6. **バイナリ名は `peercove-poc` のまま M2 を出す**。製品名へのリネームと
   crate 分割(ADR-0007 の G1b)は M3 送り(README の全手順・検証フローを
   壊す変更なので、コード署名と同時期にまとめて行う)

### 理由

- LocalSystem: wintun 自体が「SYSTEM での利用」を推奨しており、WireGuard 公式
  Windows クライアントも同構成(サービス + wintun)で動いている。Session 0 で
  懸念だった点は(a)鍵の ACL(→ 決定 3)、(b)通知(→ ADR-0009 で UI 側に
  寄せ済み)、(c)ログ(→ ADR-0009 のリングバッファが唯一の窓口として機能)
  で、いずれも設計済み・対応済み
- サービスの stderr はどこにも繋がらない(Rust はハンドル無効時に出力を黙って
  捨てる)。ログビュー(IPC のリングバッファ)を先に作っておいたことが
  サービス化の前提になっている
- ZIP を残すのは検証フローの継続性(いまの README の手順がそのまま動く)と
  上級者の需要のため。ZIP でもサービス化したい人は `service-install` を使える

### 却下案

- **ZIP のみで配布**: 起動ごとに UAC / 管理者ターミナルが必要になり、
  「CLI を使えない人でも使える」という M2 の目的に反する。受け入れ基準 2/5 も
  満たせない
- **サービスを専用ユーザーで動かす**: 権限を絞れるが、TUN 作成・ルーティング・
  フォワーディング操作に結局ほぼ管理者相当が要り、Windows では
  SeCreateSymbolicLink 等の個別権限管理が増えるだけ。M2 では LocalSystem とし、
  最小権限化は将来課題
- **sc.exe / systemctl を叩くだけの登録**(crate なし): Windows は SCM との
  状態ハンドシェイク(RUNNING/STOP_PENDING/STOPPED)をプロセス自身が実装しない
  限りサービスとして動かないので、crate は必須

### 実機 PoC で確定した Session 0 の追加要件(2026-07-09)

PoC の結果、サービス自体(SCM 登録・起動・停止・wintun のトンネル作成・
UDP 待受)は一発で動いた。ただし **Session 0 固有の「対話 UI が無い」ことに
起因する遮断が 2 つ**あり、どちらも明示的な設定が必要:

1. **秘密鍵・PSK の ACL**(決定 3 のとおり)。`host.key` だけでなく
   **PSK ファイルにも**必要(片方だけだと `preshared_key_file の読み込みに
   失敗 (os error 5)` で止まる — 実際に踏んだ)
2. **ファイアウォールの受信許可**。対話プロセスなら初回 bind 時に
   「アクセスを許可しますか?」ダイアログが出て **TCP と UDP のペア**で
   許可ルールが作られるが、Session 0 では出ない = 既定ポリシーで
   inbound が黙って落ちる。**両方**必要:
   - **UDP**(WG 待受 51820)が無い症状: 待受は正常・メンバーの tx だけ
     増えて rx 0、ハンドシェイク不成立
   - **TCP**(トンネル内コントロールチャネル 51821)が無い症状:
     ハンドシェイク・ping は通るのに**台帳だけ届かない**
     (ICMP は別ルールで許可済みだったため ping が通り、切り分けを誤りやすい)
   → `service-install` が netsh でルール(名前 `PeerCove Daemon`、
   この exe 宛の UDP と TCP のペア)を追加し、`service-uninstall` が削除する。
   **MSI(G7b)も同名ルールを install/uninstall で対に管理すること**

### 決定 5 の改訂(2026-07-09、G7b で MSI を CLI 呼び出しに変更)

決定 5 は「MSI は WiX ネイティブの `<ServiceInstall>` で登録」としていたが、
G7b 実装時に以下が判明したため、**MSI もカスタムアクションで
`peercove-poc daemon service-install` / `service-uninstall` を呼ぶ方式**に変更した
(依頼者承認。Fable への確認は `docs/peercove-g7b-msi-review-for-fable.md`):

- 上記のとおり Session 0 ではファイアウォール許可が必須だが、**Tauri 2.11 の
  WiX 設定(`bundle.windows.wix`)は任意の WiX 拡張を渡せない**(`fragmentPaths` /
  `componentRefs` / `template` / `language` 等のみ。`-ext` 相当が無い)。
  よって `WixFirewallExtension` は使えず、**どの方式でもカスタムアクションが必要**
- それなら、サービス登録もファイアウォールも一括で行う**実機検証済みの
  `service.rs`(CLI)を 1 回呼ぶ**方が、二重定義が無く堅い
- 実装は type 18(FileKey)のカスタムアクション(`daemon-service.wxs`)。
  `[INSTALLDIR]` の遅延展開・CustomActionData・WiX 拡張のいずれも不要
  (Tauri 自身のテンプレートも FileKey / `Impersonate="no"` deferred CA を使用)
- サービス名 `peercove-daemon`・引数 `daemon service`・ルール名 `PeerCove Daemon`
  は引き続き `service.rs` の定数が唯一の正本(WiX 側に文字列を持たない)

deb 側(決定 4)は変更なし(systemd unit 同梱 + maintainer script。G7b で実装)。

### 追加(2026-07-09): インストーラの日本語化とターゲット限定

- **MSI を日本語化**(依頼者要望)。`tauri.conf.json` の
  `bundle.windows.wix.language = { "ja-JP": { localePath: "windows/wix-ja-JP.wxl" } }`。
  標準ダイアログは WiX の ja-jp カルチャ(WixUIExtension)で、Tauri 独自文言
  (「Launch」「Installs …」等)は `wix-ja-JP.wxl` で上書き。生成物は
  `PeerCove_0.1.0_x64_ja-JP.msi`(ProductLanguage 1041)
- **`bundle.targets` を `["msi", "deb"]` に限定**(旧: `"all"`)。`"all"` だと
  Windows で NSIS の `-setup.exe` も生成されるが、**NSIS には
  daemon-service.wxs(WiX/MSI 専用)が効かずデーモンが登録されない**壊れた
  インストーラになる。MSI(Windows)/ deb(Linux)だけを正式ビルドとし、
  ポータブル版は `packaging/make-*.sh` で別途組み立てる

## ADR-0011: トンネル内 DNS(内蔵リゾルバ方式)(M3-1)

- **日付**: 2026-07-09(同日改訂: 複数ネットワーク対応 = ADR-0012 と、
  カスタムサブドメイン + DNS 管理画面の依頼者要望を反映)
- **ステータス**: 承認(2026-07-09)

### 背景

仮想 IP の直打ちは覚えにくく、メンバーの増減・IP 変更に弱い。依頼者要望により
`member1.peercove.internal` のようなホスト名でメンバー同士がアクセスできるように
する。方式は hosts ファイル書き換えではなく**内蔵 DNS リゾルバ**(依頼者指定)。
さらに追加要望として (a) サブドメインを追加できる作りにする、
(b) DNS 管理画面を設ける、(c) 複数ネットワーク(ADR-0012)と両立させる。

### 決定

#### 1. 名前空間: `<ラベル>.<ネットワーク名>.peercove.internal`

- `.internal` は ICANN がプライベート用途に予約した TLD(2024 年決議)。
  実在ドメインと衝突せず、所有権も不要
- **ネットワーク名がサブドメインの階層になる**(ADR-0012 の複数ネットワーク
  対応)。例: `alice.game.peercove.internal` と `alice.family.peercove.internal`
  は別ネットワークの別レコード。1 ネットワークしか使わない場合も同じ形
  (init 時の既定ネットワーク名を使う)
- メンバーのラベル部分は台帳の表示名から導出されるため、既存の rename 機能
  (UI / CLI)がそのまま DNS 名の変更になる

#### 1b. カスタムレコード(サブドメイン)と DNS 管理画面

- ホストは任意の A レコードを追加できる:
  `<任意ラベル>.<ネットワーク名>.peercove.internal → IPv4`
  例: `nas.game.peercove.internal → 100.71.42.3`(メンバーの別名)、
  将来はサブネットルーター(M3-7)配下の LAN IP も指せる
- カスタムレコードは**台帳の一部としてホストが配布**する(台帳スキーマに
  `dns_records` を追加)。メンバー側は読み取り専用
- 管理 UI: ネットワークごとの「DNS」画面。自動生成レコード(メンバー由来)は
  グレー表示で確認のみ、カスタムレコードは追加・編集・削除可(ホストのみ)。
  ラベルの検証(a-z0-9- / 63 文字 / 自動生成分との衝突)は保存時に行う

#### 2. ラベルの導出規則(台帳から決定的に)

全ノードが同じ台帳から同じ結果を得られるよう、導出は純関数にする:

1. 表示名を小文字化し、DNS ラベルとして安全な文字(`a-z0-9-`)以外を `-` に置換。
   先頭・末尾の `-` は除去、63 文字で切り詰め
2. 結果が空(日本語名など非 ASCII のみ)の場合は `member-<仮想IP第4オクテット>`
   にフォールバック(ホストは `host`)
3. 正規化後に衝突した場合は、仮想 IP が小さい方が勝ち、負けた方は
   `<ラベル>-<第4オクテット>` を付けて一意化
4. カスタムレコード(1b)と自動生成ラベルが衝突した場合は**自動生成側が勝つ**
   (メンバー名の解決が不意に奪われない)。管理画面は保存時に衝突を弾くが、
   後からメンバー名が変わって衝突した場合の優先規則としてここで確定させる
5. UI のメンバー一覧に確定した DNS 名を表示する(何と引けば届くかを見せる)

#### 3. リゾルバ: デーモン内蔵の最小 DNS サーバ

- デーモンが**自分のトンネル IP の UDP 53** で待ち受ける(デーモンは
  root/SYSTEM なので特権ポート可)。物理インターフェースには晒さない。
  複数ネットワーク稼働時は各トンネル IP で待ち、**全ネットワークのゾーンを
  合算した 1 つのテーブル**から応答する(ネットワーク名の階層で分離される
  ため衝突しない)
- 台帳(自分のローカルコピー)+ カスタムレコードを情報源に **A レコードのみ**
  応答。該当なしは NXDOMAIN、A 以外のタイプは空応答(NOERROR/NODATA)。TTL 30 秒
- **各ノードが自分で応答する**(ホストへ問い合わせない)。台帳は既に全員へ
  配布されているため、追加のネットワーク往復ゼロ・ホスト障害の影響なし
- 実装は自前(クエリ 1 問のパース + 応答組み立て、圧縮ポインタは 0xC00C 固定)。
  hickory-proto 等の crate はフル機能すぎるため採用しない(依存を増やさない方針。
  将来 AAAA/PTR が必要になったら再評価)

#### 4. OS への組み込み(スプリット DNS)

`*.peercove.internal` だけを内蔵リゾルバへ向け、他のドメイン解決には一切
干渉しない。サフィックスは常に `.peercove.internal` 全体で登録する
(ネットワークが増減しても OS 側の登録は 1 つのまま):

- **Windows**: NRPT(Name Resolution Policy Table)にルールを追加
  (`Add-DnsClientNrptRule -Namespace ".peercove.internal" -NameServers <トンネルIP>`
  相当を PowerShell 起動 or レジストリ直書きで)。最後のトンネルの `down` で必ず削除
- **Linux**: systemd-resolved に per-link 設定
  (`resolvectl dns <if> <トンネルIP>` + `resolvectl domain <if> "~peercove.internal"`)。
  リンク消滅時に resolved 側が自動でクリーンアップするが、`down` でも明示解除
- 対で実装するクリーンアップ(作業ルール)に従い、異常終了後の残骸検出も入れる

### 検討した代替案

- **hosts ファイル書き換え**: 実装は簡単だが、システムファイルを汚す・
  他ソフトとの競合・書き込み失敗時の復旧が課題。依頼者が内蔵リゾルバ方式を選択
- **ホスト集中応答(ホストの 53 番へ全員が問い合わせ)**: 実装箇所は 1 つで
  済むが、名前解決までホスト依存になり、直接通信(M3-2〜4)導入後も
  ホスト経由の往復が残る。台帳が既に分散しているのに集中させる理由がない
- **mDNS(.local)**: `.local` は既存の mDNS と衝突し、Windows/Linux で挙動が
  不安定になる既知の罠。不採用

### 影響

- `peercove-core`: ラベル導出の純関数(+ユニットテスト)、台帳スキーマに
  `dns_records`(カスタムレコード)を追加
- `peercove-ops`: カスタムレコードの追加・編集・削除(設定ファイル操作)
- `peercove-poc`: DNS サーバタスク(tokio)、OS 設定の trait 追加
  (`DnsBackend` 的な薄い抽象。バックエンドは `#[cfg(target_os)]` で分岐)
- UI: メンバー一覧に DNS 名表示 + ネットワークごとの DNS 管理画面
- 実装順: ADR-0012(複数ネットワーク)を先に実装し、その上に DNS を載せる
  (ネットワーク名の階層・管理画面のスコープが最初から複数前提になるため)

## ADR-0012: 複数ネットワークの同時参加・同時ホスト(M3-0)

- **日付**: 2026-07-09
- **ステータス**: 承認(2026-07-09)

### 背景

M2 handoff Q4 では「同時参加は 1 ネットワークまで」としていたが、依頼者要望に
より複数ネットワークの同時運用を可能にする(例: A+B のネットワークと A+C の
ネットワークを並行運用)。同一マシンが「ネットワーク X のホスト」かつ
「ネットワーク Y のメンバー」になるケースも許す。

### 決定

#### 1. ネットワークは「名前」を持つ第一級の概念になる

- `init` 時にネットワーク名を指定(UI では入力欄、既定は `home` などの提案値)。
  名前は DNS のサブドメイン階層(ADR-0011)と設定ディレクトリ名に使うため、
  ラベル導出規則(ADR-0011 §2)で正規化したスラッグを内部 ID とする
- 招待トークンにネットワーク名を含める(メンバー側が join したとき、どの
  ネットワークかが分かる)。既存フィールドの末尾に追加し、**旧トークンは
  名前なしとして受理**(既定名 `home` を割り当て)。
  **実装(M3-0a)**: 名前ありはトークン内部バージョン 2、名前なし(既定名)は
  v1 のままエンコードするため、既定名運用なら旧バイナリでも読める

#### 2. 設定の配置: `<設定ディレクトリ>/networks/<スラッグ>/`

- 各ネットワークのディレクトリに設定と鍵(+PSK)を置く。
  **実装(M3-0a)**: ファイル名は従来どおり host.toml / member.toml を踏襲し、
  どちらが存在するかで役割を判別する(config.toml へ改名しない —
  移行が純粋なファイル移動で済み、鍵の相対参照も壊れないため)
- 既存の固定スロット(host.toml / member.toml)は、デーモン or UI の初回起動時に
  `networks/` へ**自動移行**する(バックアップを残す)。CLI の `--config` 直指定は
  従来通りどのパスでも動く
- ネットワークの削除 = ディレクトリの削除(UI から可能にする。稼働中は不可)

#### 3. 衝突回避

- **サブネット**: 起動(up)時に、稼働中の他ネットワークとサブネットが重なる
  場合は明確なエラーで拒否(init のランダム /24 = ADR-0006 で実用上ほぼ回避
  できるが、黙って壊れない)
- **待受ポート**: ホストを兼ねる場合はネットワークごとに listen_port が必要。
  init 時に既存ネットワークを見て未使用ポート(51820, 51821, …)を自動選択
- **コントロールチャネル**: 現行の TCP 51821 は**トンネル内アドレス**で待つため、
  ネットワークごとにトンネル IP が異なれば同一ポートのままで衝突しない
  (バインド先をワイルドカードでなく各トンネル IP に固定する)
- **インターフェース名**: `peercove0`, `peercove1`, …を空き番号で採番。
  **実装(M3-0b)**: Windows も同じ採番方式に統一(当初案の
  `PeerCove <スラッグ>` は、スラッグ変更・再参加でアダプタ名が揺れて
  wintun のアダプタ再利用と相性が悪いため不採用)。設定で明示した
  インターフェース名が稼働中と重複した場合は自動で別名にせずエラー
- **Windows ファイアウォール**: サービスの受信許可ルールは
  `program=peercove-poc.exe` ベース(ポート非依存)のため、
  多重ホストで待受ポートが増えても追加設定は不要(M2-G7a の設計が
  そのまま効く)

#### 4. デーモンと IPC

- `DaemonShared` の `Option<Active>` を `HashMap<スラッグ, Active>` にする。
  start / stop / status はネットワーク指定に変わる(status は全ネットワーク分を
  返す)。IPC スキーマの破壊的変更になるが、デーモンと UI/CLI は同梱配布なので
  互換レイヤは設けない(ADR-0007 の方針を踏襲)
- UPnP・ファイアウォール・フォワーディングなどの OS 資源はネットワーク単位で
  取得・解放し、`down` の対称性を維持する

#### 5. UI

- スタート画面を**ネットワーク一覧**に変える。各ネットワークはカードで
  「名前・役割(ホスト/メンバー)・状態・メンバー数」を表示し、個別に
  接続/切断できる。「+ ネットワークを追加」から新規ホスト or 招待で参加
- 詳細画面(現在の TunnelView 相当)はカードから遷移。招待・メンバー管理・
  DNS 管理(ADR-0011)・設定はネットワークごとのスコープになる

### 検討した代替案

- **1 トンネルに複数サブネットを載せる**: WG 的には AllowedIPs で可能だが、
  「A+B」「A+C」のような**参加者の分離**(B から C が見えない)が保てない。不採用
- **デーモンをネットワークごとに複数起動**: プロセス管理・IPC の多重化・
  サービス登録が複雑化する割に利点がない。単一デーモンの多重トンネルとする

### 影響と実装順(M3-0 を 3 ゴールに分割)

1. **M3-0a**: peercove-ops の設定配置刷新(networks/ + 自動移行)+
   ネットワーク名(init・トークン拡張)
2. **M3-0b**: デーモンの多重トンネル化 + IPC 改訂 + CLI 追随
3. **M3-0c**: UI のネットワーク一覧化
4. その後に M3-1(DNS)を複数ネットワーク前提で実装する

## ADR-0013: メンバー間直接通信(UDP ホールパンチング)(M3-2〜4)

- **日付**: 2026-07-10
- **ステータス**: 承認(2026-07-10 依頼者承認。以下の追加条件つき)

**依頼者の追加条件(2026-07-10)**:

1. **エンドポイントに観測時刻を持たせ、古いエンドポイントは直接試行しない**。
   実装: マシン間の時計ずれを避けるため絶対時刻ではなく
   `endpoint_age_secs`(ホストが観測してからの経過秒)を台帳に載せる。
   ホスト側でも新鮮なもの(最終ハンドシェイクがオンライン判定閾値内)しか
   載せず、メンバー側は「受信時の age + 受信からの経過」が閾値を超えたものを
   試行対象から外す(二重ガード)
2. **「直接通信を試す」の ON/OFF をユーザー設定として将来用意できるようにする**。
   実装: member 設定に `direct = true|false`(既定 true)を M3-3 で追加し、
   直接ピア管理ロジックの入口を単一フラグで制御する。UI のトグルは
   将来ゴール(設定画面)で公開する
3. **外部 IP:port がネットワーク内の全員に共有される点を README と UI 上の
   説明に明記する**(UI は経路表示を追加する M3-4 で説明文を入れる)
4. **Windows の経路選択(最長プレフィックス一致)は M3-3 の前に修正する**
   (M3-2 で実施)

### 背景

Phase 1 はハブ&スポーク構成で、メンバー間の通信はすべてホストを経由する。
ゲームや大容量転送では「メンバー A → ホスト → メンバー B」の 2 ホップ分の
遅延と、ホスト回線の帯域がボトルネックになる。roadmap トラック B は
メンバー間で WG トンネルを直接張り(NAT 越えは UDP ホールパンチング)、
失敗時はホスト経由へフォールバックする、という段階的な計画になっている。

前提となる観測事実:

- ホストは各メンバーの**外部エンドポイント**(NAT 変換後の IP:port)を既に
  知っている(WG の `PeerStats.endpoint`。メンバーが keepalive を送るたびに
  最新化される)
- メンバーの WG は 1 つの UDP ソケットで全ピアと通信する。フルコーン/
  アドレス制限/ポート制限コーン NAT では「ホスト向けに開いたマッピング」が
  他の宛先にもそのまま使えるため、**ホストが見ているエンドポイント宛に
  双方から送り合えば穴が開く**(WireGuard 標準のハンドシェイク再送 +
  keepalive がパンチ動作を兼ねる)
- シンメトリック NAT は宛先ごとにマッピングが変わるため、この方式では
  直接接続できない → フォールバックで救う

### 選択肢

パンチングの調停(両側が「いつ・誰に」穴を開けるかの同期)について:

1. **台帳駆動(暗黙の同時パンチ)**: 台帳(`LedgerEntry`)に各メンバーの
   外部エンドポイントを載せて配布し、各メンバーは決定的ルール
   (「オンラインでエンドポイント付きの他メンバー全員に直接ピアを張る」)で
   自律的に WG ピアを追加する。台帳の再配布が事実上の同期信号になる
   (変更時に全員へ数秒内に届く)。**新しい制御メッセージは不要**
2. **明示的調停**: ホストが `PunchStart` を両者へ送って開始タイミングを
   合わせ、成否を報告させる。同期は厳密になるが、状態機械とメッセージが
   コントロールチャネルに増える
3. STUN/ICE 相当の候補収集(LAN アドレス含む複数候補を試す): Tailscale 方式。
   同一 LAN 内のメンバー同士も最短経路になるが、実装規模が大きい

### 決定

**選択肢 1(台帳駆動)を採用**する。実機検証で「双方の追加タイミングのずれ」が
実際に問題になった場合のみ、選択肢 2 を追加する。選択肢 3 の LAN 候補は
将来拡張(このADRの方式はそのまま土台になる)。

#### 1. エンドポイントの学習と配布(B-1 = M3-2)

- ホストは supervise の周期(5 秒)で `PeerStats.endpoint` を読み、
  `LedgerEntry` に追加する新フィールド
  `endpoint: Option<SocketAddr>`(`skip_serializing_if = None`)に載せる
- 旧メンバーは未知フィールドとして無視、旧ホストからは届かない(= None)。
  **`PROTO_VERSION` は据え置き**(Ledger への追加フィールドは M3-1 と同じ互換規則)
- ホスト自身のエントリにはエンドポイントを載せない(メンバーは設定で
  ホストの endpoint を既に持っている)
- **プライバシー上の注意**: メンバーの外部 IP:port が同一ネットワークの
  全員に見えるようになる(Tailscale 等と同等。招待制の閉じたネットワーク
  なので許容)。README に明記する

#### 2. 直接ピアの追加(B-2 = M3-3)

- メンバーは台帳を受け取るたびに「望ましい直接ピア集合」を計算する:
  **自分とホストを除く、オンラインかつエンドポイント付きの全メンバー**
- 各相手について実行中の WG へ**ランタイムでピアを追加**する
  (`add_peer`。設定ファイルには書かない — エフェメラルな最適化であり、
  正本の台帳から毎回導出できるため)。パラメータ:
  - `allowed_ips = [相手の仮想 IP /32]`
  - `endpoint = 台帳のエンドポイント`
  - `persistent_keepalive = 25`(NAT マッピング維持 + パンチの再送)
  - **PSK なし**。ペア PSK の生成・配布はコントロールチャネル経由で可能だが、
    秘密材料の配布経路と保管が増える割に、WG の Noise ハンドシェイクで
    機密性・認証は既に担保されている(PSK は耐量子の上乗せ)。
    ホスト↔メンバーの既存 PSK は従来通り。M3-11(鍵ローテーション)で再検討
- 双方が同じ台帳から同じ結論に達するので、両側がほぼ同時にピアを追加し、
  WG のハンドシェイク再送(5 秒間隔)が双方向に走って穴が開く
- **経路選択は WG の cryptokey routing の最長一致**に任せる:
  ホストピアの `/24` より直接ピアの `/32` が勝つ。直接ピアを消せば
  自動的に `/24`(ホスト経由)へ戻る。OS ルート・フォワーディング設定は
  一切変更しない(ホストのリレーは常に生きている)
- **前提修正(Windows)**: 自作デバイスの `find_peer_by_dst` は現在
  first-match(HashMap 順で不定)のため、**最長プレフィックス一致に修正する**。
  Linux はカーネル WG がネイティブに最長一致(trie)なので変更不要

#### 3. フォールバックと経路の可視化(B-3 = M3-4)

- 直接ピアごとに状態機械を持つ:
  - **確立中(Trying)**: 追加直後。30 秒以内に最終ハンドシェイクが
    観測できなければ失敗と判定してピアを削除 → 中継(Relay)へ
  - **直接(Direct)**: 最終ハンドシェイクが新しい(180 秒以内 = WG の
    セッション上限)。陳腐化したら削除して中継へ
  - **中継(Relay)**: 直接ピアなし(= 従来のホスト経由)。バックオフ付きで
    再試行する(初回失敗後 5 分、以後 2 倍で上限 1 時間。台帳の
    エンドポイントが**変わったら即時**再試行)
- 判定は**各メンバーのローカル観測のみ**で行う(調停メッセージなし)。
  片側だけ直接ピアを持つ非対称状態は一時的なブラックホールになり得るが、
  ハンドシェイクが完了しない限り 30 秒で必ず中継へ収束する
- status / UI にメンバーごとの経路を表示する:
  「直接」「中継(ホスト経由)」「確立中」。IPC の `TunnelInfo` に追加
  フィールドで載せる(旧互換は M3-0b と同じ追加フィールド規則)
- 既知の制約(README に記載): シンメトリック NAT 同士は直接化できない/
  同一 LAN 内のメンバー同士は NAT ヘアピン非対応ルーターだと失敗する
  (いずれも中継で通信自体は維持される)

### 理由

- 新しいプロトコルメッセージなしで成立する(台帳の追加フィールド 1 つ)。
  コントロールチャネルの互換規則(M3-1 で確立)にそのまま乗る
- 「直接ピアの追加/削除」が唯一の可変要素で、失敗時の回復が
  「ピアを消すだけ」— down の対称性・クリーンアップの原則に合う
- WG 自体のハンドシェイク再送をパンチに流用するため、タイミング同期の
  精度要求が低く、選択肢 2 の複雑さを(少なくとも初版では)回避できる

### 実装順(3 ゴール)

1. **M3-2(B-1)**: Windows デバイスの最長一致修正(追加条件 4)+
   `LedgerEntry.endpoint` / `endpoint_age_secs` + ホストの学習・配布。
   ここまでは挙動が何も変わらない(情報が増えるだけ)
2. **M3-3(B-2)**: メンバーの直接ピア追加/削除ロジック
   (`direct` 設定フラグ + 鮮度ガード + 確立中タイムアウトでの除去 = 安全弁)。
   パンチ成功の実機検証はここで行う
3. **M3-4(B-3)**: 状態機械の仕上げ(バックオフ付き再試行)+
   status / UI の経路表示(+ 外部 IP 共有の説明文)

## ADR-0014: サブネットルーター(メンバー背後 LAN への経路広告と中継)(M3-7)

- **日付**: 2026-07-10
- **ステータス**: 承認(2026-07-10 依頼者承認。ルーター役は V1 Linux 限定で可・
  SNAT 既定で可。**追加条件: 上級者向けに「SNAT なし(LAN 側静的ルート前提)」
  モードを将来用意できる構造にする** — ルーター役の転送セットアップは
  SNAT 有無をフラグで受け、将来ルーター役側の設定
  (例 `[interface] subnet_snat = false`)で切れるようにしておく)

### 背景

PeerCove を入れられない LAN 機器(NAS・プリンタ・組込み機器など)へ、他の
メンバーからトンネル越しに届きたい。あるメンバーを「ルーター役」にして、
その背後のサブネット(例 `192.168.10.0/24`)への経路を全員に配る機能
(Tailscale の subnet router 相当)。

現状の前提:

- 経路は仮想サブネット(/24)の connected route だけで成立しており、
  仮想サブネット外の allowed_ips は「未検証の構成」として警告している
- Windows ホストのメンバー間リレーは M3-2 で最長プレフィックス一致に
  なっており、サブネット宛の転送に対応できる下地がある
- カスタム DNS レコード(M3-1c)は任意の IPv4 を登録できるため、
  NAS に `nas.<ネットワーク名>.peercove.internal` と名前を付ける用途に
  そのまま使える(変更不要)

### 選択肢

**1. 広告(どのメンバーがどのサブネットを持つか)の正本をどこに置くか**

- A: ルーター役メンバーが自分の設定で自己申告し、ホストが承認する
  (Tailscale 方式)。承認の保留状態と制御メッセージが増える。
  未承認の広告を無視するガードも必要
- B: **ホストの host.toml の `[[peer]]` に `subnets = […]` を書く
  (ホストが正本)**。台帳で全員に配布し、ルーター役自身も台帳の
  自分のエントリを見て「自分がルーターであること」を知る。
  承認フローは不要(書けるのがホスト管理者だけなので)

**2. 戻り経路(LAN 機器 → メンバーへの応答)をどう成立させるか**

- LAN 側のルーターに静的経路(仮想サブネット → ルーター役)を書いてもらう。
  NAT なしで送信元 IP がそのまま見えるが、ユーザーの手作業が増え、
  家庭用ルーターでは設定できない機種もある
- **ルーター役で SNAT(MASQUERADE)する**。LAN 側は設定不要で必ず動く。
  LAN 機器からは全アクセスがルーター役の LAN IP に見える(制約として明記)

### 決定(提案)

**1-B(ホスト設定が正本)+ AllowedIPs / OS ルート + SNAT** を採用する。

1. **設定**: host.toml の `[[peer]]` に `subnets = ["192.168.10.0/24"]` を
   追加(serde は default + 空なら省略)。検証: 仮想サブネットとの重複禁止・
   ピア間の重複禁止・プレフィックスは /8〜/32・RFC1918 外は警告。
   注意: 設定は `deny_unknown_fields` のため、**subnets を書いた host.toml は
   旧デーモンでは読めない**(黙って無視されるより安全な、明示エラーになる)
2. **台帳**: `LedgerEntry` に `subnets: Vec<Ipv4Net>`(default +
   空なら省略)。**PROTO_VERSION は据え置き**(ADR-0013 と同じ
   追加フィールド互換規則 — 旧バージョンとは互いに無視し合う)
3. **ホスト**: ルーター役ピアの AllowedIPs を `[仮想IP/32] + subnets` にする
   (既存 sync_peers の指紋比較で動的反映)。ホスト自身が LAN 機器へ届く
   よう OS ルート(subnet → トンネル IF)も追加する。メンバー間リレーは
   Windows = デバイス内最長一致(OS ルート不要)、Linux = per-IF
   forwarding(既存)+ 上記 OS ルートで成立する
4. **メンバー(利用側)**: 台帳に他ピアの subnets を見つけたら、ホストピアの
   AllowedIPs に加えて再登録し、OS ルートを追加する。撤回されたら対で削除。
   **V1 ではサブネット宛は常にホスト経由**(直接ピア(/32)には載せない。
   直接経路へのサブネット追加は将来拡張)。`direct` 設定とは独立に機能する
5. **ルーター役**: 台帳の自分のエントリに subnets があれば転送を有効化する。
   - **Linux**: トンネル IF と LAN 側 IF(サブネットへの経路から特定)の
     per-IF forwarding を有効化 + `iptables -t nat` の MASQUERADE
     (送信元 = 仮想サブネット、宛先 = 広告サブネットに限定)。down で対削除。
     iptables が無い場合は分かりやすいエラー
   - **Windows**: **V1 ではルーター役は非対応**(利用側としては対応)。
     WinNAT / ICS の制約調査が必要なため将来ゴールに切り出す
6. **WgBackend**: OS ルート操作を trait に追加
   (`add_route` / `remove_route`。Linux = `ip route replace/del … dev`、
   Windows = `netsh interface ipv4 add/delete route`)。トンネル IF の消滅で
   OS が経路も消すため、down での明示削除は保険
7. **CLI / UI**: CLI に `subnet set <メンバー> <CIDR…>` / `subnet clear`
   (peercove-ops 拡張。ホスト設定の編集なので非特権)。UI はホストの
   メンバー行に「サブネット」編集を追加し、全員のメンバー一覧に
   広告サブネットのバッジを表示する

### セキュリティ・制約(README に明記する)

- 経路の正本がホスト設定なので、**メンバーが勝手に広告して他メンバーの
  トラフィックを吸い取ることはできない**
- ルーター役の背後 LAN は(サブネット単位で)全メンバーから到達可能になる。
  メンバー単位の絞り込みは ACL(M3-10)まで無い
- SNAT のため、LAN 機器から見るとアクセス元はすべてルーター役の端末になる
- メンバー自身の LAN と広告サブネットのレンジが重複していると、その
  メンバーからは届かない(ローカルの connected route が優先される)

### 理由

- ホスト正本なら承認プロトコルが不要で、プロトコル変更が台帳の追加
  フィールド 1 つに収まる(ADR-0013 で実証済みの互換規則に乗る)
- SNAT なら LAN 側の設定変更ゼロで確実に動く(NAS ユースケースの体験優先)。
  NAT なし運用が必要になったら「SNAT を切る設定 + LAN 側静的経路」を
  後から足せる
- 失敗時の回復が「ルートと AllowedIPs を戻すだけ」で、down の対称性
  (クリーンアップの原則)を保てる

### 実装順(2 ゴール)

1. **M3-7a**: コア + デーモン(config `subnets` + 検証 → 台帳配布 →
   ホストの AllowedIPs/OS ルート → メンバーの AllowedIPs/OS ルート →
   Linux ルーター役の forwarding + SNAT → CLI)。実機検証はここで
   (Linux メンバーの背後に疑似 LAN を作って確認)
2. **M3-7b**: UI(サブネット編集 + バッジ + 説明文)+ README 検証手順

### 将来課題

- Windows がルーター役になる対応(WinNAT の調査が前提)
- サブネット宛の直接経路(直接ピアの AllowedIPs への追加)
- exit node(`0.0.0.0/0` の広告)は用途も影響も別物なので、やるなら別 ADR

## ADR-0015: トンネル内メッセージング基盤(ファイル送信・チャット)(M3-9 / M3-13)

- **日付**: 2026-07-10
- **ステータス**: 承認(2026-07-10 依頼者承認。P2P 方式・自動受信+受信ボックス・
  オフライン宛は V1 非対応(オンラインのみ)、いずれも提案どおりで確定)

### 背景

M3-9(ファイル送信、Taildrop 相当)と M3-13(チャット、LINE 風の 1:1 /
ネットワーク全体)は、どちらも「メンバー同士がトンネル内でアプリデータを
やり取りする」機能で、基盤を共有する前提(roadmap)。対象は VPN 内メンバー
のみ・外部サーバー不要・履歴はローカル保存、が依頼者要件。

現状の持ち物:

- コントロールチャネル(ホスト仮想 IP の TCP 51821、JSON Lines)は
  **ホスト↔メンバーのスター型**。メンバー同士は直接会話できない
- トンネル自体はメンバー間の任意の IP 通信を運べる(ハブ&スポーク中継、
  M3-3 以降は条件が揃えば直接経路)。送信元仮想 IP は WG の cryptokey
  routing で偽装不能 = **仮想 IP がそのまま送信者の身元になる**
- デーモンは常駐(UI が閉じていても受信できる)。UI は IPC ポーリングで
  状態を取る(ADR-0009 のパターン)

### 選択肢

**1. トランスポート(メッセージをどの経路で運ぶか)**

- A: コントロールチャネルに載せる(ホスト経由リレー)。
  既存接続を再利用でき、ホスト預かり(オフライン宛の再配送)も作りやすい。
  ただしファイルを JSON Lines + base64 で運ぶのは非効率で、ホストの
  アプリ層が全メッセージを解釈・中継する分だけ負荷と実装が増える。
  直接経路(M3-3)があっても使われない
- B: **各デーモンが自分の仮想 IP で メッセージ用 TCP(51822)を待ち受け、
  送信側が相手の仮想 IP へ直接接続する(P2P)**。経路選択は IP 層に任せる
  (ハブ&スポーク中継でも直接経路でも同じコードで動き、直接経路なら
  ファイル転送がその帯域・レイテンシの恩恵を受ける)。ホストは
  IP パケットを転送するだけでアプリ層の関与なし。ホスト自身も
  1 メンバーとして同じリスナーを持つ

**2. フレーミング**

- コントロールチャネルと同じ **JSON Lines を制御フレームに使い、
  ファイル本体だけ生バイナリをそのまま流す**(HTTP のヘッダ+ボディと
  同じ発想)。1 論理操作 = 1 短命 TCP 接続(チャット 1 通、ファイル 1 個)。
  常設接続の管理(keep-alive・再接続・take 上限の罠)を持ち込まない

**3. 受信側の認証**

- リスナーは仮想 IP のみに bind し、**接続元 IP が台帳のメンバー仮想 IP で
  ある場合だけ受け付ける**。広告サブネット(ADR-0014)内の LAN 機器や
  ルーター役背後からの接続は台帳に無い IP なので自動的に拒否される。
  追加の暗号は載せない(各ホップは WG が暗号化。ハブ&スポーク経由は
  ホストで復号・再暗号化されるが、これは全トンネル通信と同じ性質)

### 決定(提案)

**1-B(P2P、各デーモンの TCP 51822 待受)** を採用し、次の形で共有基盤を作る。

1. **プロトコル**(core に `msg.rs` 新設、`MSG_PORT = 51822`、
   `MSG_VERSION = 1`)。接続ごとに `hello`(version + 自分の公開鍵)を
   送ってから本題:
   - チャット: `chat { id, scope: "direct" | "network", text, sent_at }`
     → 受信側が `ack { id }`。ネットワーク全体宛は**送信側がオンライン
     メンバー全員に個別送信**(scope で「全体」と分かるようにする)
   - ファイル: `file_offer { id, name, size }` → `file_accept` /
     `file_reject { reason }` → 生バイト列 → `file_hash { sha256 }`
     (送信側が転送しながら計算 — 巨大ファイルを 2 回読まない)→
     受信側が検証して `file_done`。V1 は途中再開なし(失敗したら最初から)
2. **受信ファイルの置き場**: デーモンが
   `networks/<ネットワーク>.inbox/` に保存(受信ボックス。書きかけは
   `.part`、確定後に隣へ送信者情報の `.pcvmeta` を置く)。UI の「保存」は
   任意の場所へのコピー + スプール削除で、**UI が直接ファイル操作する**
   (IPC には乗せない — ADR-0007 と同じ整理。特権デーモンが作る
   ディレクトリは、IPC ソケットを 0666 にするのと同じ割り切りで
   非特権 UI から読み書きできる権限にする)。デーモンがユーザーの
   Downloads へ直接書かないのは所有権の問題を避けるため
3. **チャット履歴**: デーモンが `networks/<ネットワーク>.chat.jsonl` に
   追記(status ファイルと同じ配置規則)。上限(例 10,000 通)を超えたら
   古い方から捨てる。**履歴は各端末のローカルのみ**(他端末との同期なし)
4. **IPC 追加**(ADR-0007 の枠内): `SendFile`(ローカルパス指定、
   応答は転送 id)/ 進捗は Status 応答の `TunnelInfo.transfers` に載せる
   (新しいポーリング経路を作らない)。チャットは `ChatSend` /
   `ChatHistory`(M3-13)。UI は既存どおりポーリングで新着を検知して
   通知する
5. **互換性**: 旧デーモンにはリスナーが無く接続拒否になるだけ
   (コントロールチャネル・台帳には一切手を入れない = PROTO_VERSION
   据え置き)。UI は「相手のバージョンが未対応」と表示する
6. **オフライン宛**: V1 は**オンラインのメンバーにのみ送れる**
   (オフライン宛はチャット = 送信不可表示、ファイル = 選択不可)。
   ホスト預かり(store & forward)は将来課題
7. **秘匿ルール**: チャット本文・ファイル名はログに出さない
   (トークン等と同じ扱い)

### セキュリティ・制約(README に明記する)

- 送信者の身元は WG の cryptokey routing が保証する仮想 IP + 台帳照合。
  なりすましには WG 秘密鍵の窃取が必要(= トンネル自体の前提と同じ)
- ハブ&スポーク経由のメッセージ/ファイルはホストで復号・再暗号化される
  (ホスト管理者は技術的には読める)。直接経路(M3-3)成立時は E2E
- 受信ボックスは自動受信のため、メンバーは相手の同意なくファイルを
  置ける(容量を消費する)。対策として**受信サイズ上限**を設ける
  (既定 100 MB。受け取る側の設定 `[interface] max_recv_file_mb`、
  0 で無制限 — 2026-07-11 依頼者指定)。メンバー単位の制御は
  ACL(M3-10)まで無い
- ネットワーク全体宛チャットは「送信時にオンラインだったメンバー」に
  しか届かない(オフラインだったメンバーは受け取れない)

### 理由

- P2P なら M3-3 の直接経路がそのまま効き、ホストに新しい責務を足さない
  (「ホストはコーディネーター兼 IP リレー」という現行の役割分担を守る)
- 1 操作 = 1 接続の JSON Lines + 生ボディは、コントロールチャネルで
  実証済みの道具立てで済み、ADR-0009 で踏んだ常設接続の罠を避けられる
- 受信ボックス方式は「UI が閉じていても受信できる」「特権デーモンが
  ユーザー領域に書かない」を同時に満たす

### 実装順(4 ゴール)

1. **M3-9a**: 基盤 + ファイル送信(core `msg.rs` → デーモンのリスナー +
   送信タスク + 受信ボックス → IPC → CLI `send-file`(検証用))
2. **M3-9b**: ファイル送信 UI(メンバー行から送信・進捗・受信ボックス +
   通知)+ README 検証手順
3. **M3-13a**: チャット(chat フレーム + 履歴保存 + IPC)
4. **M3-13b**: チャット UI(タブ追加、1:1 / 全体の切替、未読バッジ、
   LINE 風の吹き出し)+ README 検証手順

(M3-13 の詳細設計と実装順は ADR-0016 で拡張・更新)

### 将来課題

- オフライン宛のホスト預かり(store & forward)と既読・配達確認
- ファイル転送の途中再開(レジューム)
- ~~任意グループ(1:1 とネットワーク全体の間)のチャット~~
  → ADR-0016 で V1 対応に昇格(2026-07-11 依頼者要望)

## ADR-0016: チャットの詳細設計(グループ・チャット内ファイル送信)(M3-13)

- **日付**: 2026-07-11
- **ステータス**: 承認(2026-07-11 依頼者承認。グループ = P2P 伝搬方式、
  V1 のグループ操作 = LINE 同等(作成・追加・退出・改名)、いずれも提案どおり)

### 背景

ADR-0015 承認時のチャットは 1:1 とネットワーク全体のみで、任意グループは
将来課題だった。M3-9(ファイル送信)完了後、依頼者から追加要望
(2026-07-11):

1. 特定のメンバーだけのグループチャットを作れるようにする
2. UI は LINE の作りに寄せる
3. チャットからファイル送信もできるようにする
4. ドラッグ&ドロップでファイル送信できるとうれしい

サーバーレス前提なので「グループ」という共有状態をどこに置くかが論点。

### 選択肢

**1. グループ状態の持ち方**

- A: **P2P 伝搬**。グループ情報(全量)を `group_update` フレームで
  メンバー同士が直接配り、各端末がローカル保存。ホストに新しい役割を
  足さず、ホスト停止中でも作成・変更できる。弱点はオフラインだった
  メンバーへの伝搬遅延(下記の自動再送でカバー)
- B: ホスト管理。台帳にグループを載せてコントロールチャネルで配布。
  整合性は確実だが PROTO_VERSION 変更(全端末更新必須)が要り、
  ホスト停止中はグループ作成・変更ができない

**2. V1 のグループ操作範囲**

- LINE 同等(作成・メンバー追加・自分の退出・改名)。プロトコル上は
  どの操作も「グループ全量 + リビジョン」の伝搬で同じなので、
  作成のみに絞ってもコスト差が小さい → LINE 同等とする

### 決定

**1-A(P2P 伝搬)** を採用。ADR-0015 の基盤(TCP 51822、JSON Lines)に
以下を足す。

1. **スコープ**: 会話は `direct`(1:1)/ `network`(全体)/
   `group`(任意グループ)の 3 種。チャットフレームは
   `chat { id, scope, group_id?, text, sent_at }` → `chat_ack { id }`。
   network / group 宛は**送信側がオンラインの対象メンバー全員に個別送信**
2. **グループ**: `{ id, name, members: [仮想IP], revision, updated_by }`。
   ローカル保存は `networks/<ネットワーク>.groups.json`(status ファイルと
   同じ配置規則)。作成・追加・退出・改名はいずれも
   `group_update { group(全量) }` を対象メンバー全員に送信。受信側は
   revision が手元より新しければ置換(同値は updated_by の IP 比較で決着 =
   最新リビジョン勝ち。同時編集は片方が消えるが、規模的に許容)。
   自分が members に居ない update を受けたら「退出/除外された」として
   会話リストから隠す(履歴はローカルに残す)
3. **オフラインメンバーへの追いつき**: デーモンはピアの
   オフライン→オンライン遷移を検知したら、そのピアが属する既知グループの
   `group_update` を送り直す(作成時にオフラインだったメンバーも、誰かと
   同時にオンラインになれば自動で追いつく)
   (**実装時修正 2026-07-11**: 遷移検知でなく **ack ベースの再送**にした。
   オンライン判定には最終ハンドシェイクの猶予(180 秒)があり、短時間の
   オフライン→オンラインが遷移として見えず追いつきが発火しない事故が
   実機検証で起きたため。各メンバーはグループごと・相手ごとに ack 済み
   revision を覚え、未達のオンラインメンバーへ 30 秒間隔で送り直す)
4. **履歴**: ADR-0015 どおり `networks/<ネットワーク>.chat.jsonl` に一本化
   (scope / group_id / kind で区別、上限 10,000 通)。各エントリに
   デーモンが単調増加の `seq` を振り、`TunnelInfo` に最新 `chat_seq` を
   載せる。UI は seq が進んだときだけ `ChatFetch { after_seq }` で差分を
   取る(毎回全履歴を読まない・新しいポーリング経路を作らない)
5. **チャット内ファイル送信**: `file_offer` に**任意フィールド**
   `chat { scope, group_id? }` を追加(serde は未知フィールドを無視する
   ので M3-9a デーモンとも相互運用可 — 旧相手にはチャット文脈なしの
   通常ファイル受信になるだけ)。チャット文脈付きの転送は送受両側が
   kind = file の履歴エントリ(name / size / 転送 id)を記録し、UI は
   会話内にファイルバブル(名前・サイズ・進捗・保存)を出す。
   実体は従来どおり受信ボックス(保存・削除の仕組みは M3-9b を再利用)。
   group / network 宛ファイルは対象メンバーへの個別転送(履歴は 1 エントリ)
6. **IPC 追加**(ADR-0007 の枠内、IPC_VERSION 据え置き):
   `ChatSend { config, scope, peer?, group_id?, text }` /
   `ChatFetch { config, after_seq }`(応答: messages + groups + 最新 seq)/
   `GroupCreate { name, members }` / `GroupUpdate { id, name?, add? }` /
   `GroupLeave { id }`。既存 `SendFile` にチャット文脈パラメータを追加
   (**実装時修正 2026-07-11**: groups は ChatFetch 応答でなく
   `TunnelInfo.groups` に載せた。チャットが無いグループ変更(改名など)は
   chat_seq が進まず UI が ChatFetch しないため、status ポーリングに
   載せる方が確実で、追加のポーリング経路も要らない)
7. **UI(LINE 風)**: チャットタブ = 会話リスト(全体・グループ・
   メンバー 1:1。最終メッセージプレビュー + 未読バッジ)→ 会話画面
   (自分は右・相手は左にアバター付き吹き出し、日付区切り、下部に
   入力欄 + 📎)。未読管理は UI ローカル(会話ごとの最終既読 seq を
   localStorage)。**ドラッグ&ドロップ**は Tauri のネイティブファイル
   ドロップイベント(webview の drag-drop)で受け、会話画面への
   ドロップ → 確認 → その会話の宛先へ送信
8. **秘匿ルール**: チャット本文・グループ名・ファイル名はログに出さない
   (id・IP・サイズは可)

### 制約(README に明記する)

- グループの整合性は最新リビジョン勝ち。同時に別々の変更をすると
  片方が失われる(小規模グループ前提の割り切り)
- オフラインだったメンバーは、自分がオンラインのときに他のメンバーが
  オンラインにならないとグループ変更・メッセージを受け取れない
  (V1 はオンライン配送のみ — ADR-0015 と同じ)
- 履歴・グループ情報は端末ローカル(端末間同期なし)

### 理由

- P2P 伝搬は「ホストはコーディネーター兼 IP リレー」という役割分担を
  守り、PROTO_VERSION・コントロールチャネルに手を入れない
  (旧デーモン相手でも壊れない)
- グループ操作を全量+リビジョンの置換に統一すると、作成・追加・退出・
  改名が 1 つのフレームで済み、部分更新の順序問題を持ち込まない
- ファイルバブルは M3-9 の転送・受信ボックスをそのまま再利用し、
  チャット文脈は任意フィールド 1 つの追加で済む

### 実装順(4 ゴール — ADR-0015 の 13a/13b を置き換え)

1. **M3-13a**: チャット基盤(chat / chat_ack フレーム、履歴 jsonl + seq、
   IPC ChatSend / ChatFetch、CLI 検証コマンド)。direct / network のみ
2. **M3-13b**: チャット UI(LINE 風の会話リスト + 吹き出し、1:1 / 全体、
   未読バッジ、OS 通知)+ README 検証手順
3. **M3-13c**: グループ(group_update フレーム + groups.json + 追いつき
   再送 + IPC + UI の作成/追加/退出/改名)+ README 検証手順
4. **M3-13d**: チャット内ファイル送信 + ドラッグ&ドロップ(file_offer の
   chat 文脈、ファイルバブル、D&D)+ README 検証手順

### 将来課題

- 既読表示・配達確認(ホスト預かりと合わせて ADR-0015 の将来課題)
- グループのアイコン・メンバー権限(管理者のみ追加可、など)

## ADR-0017: チャットのリンクプレビュー(OGP、各端末が取得)(M3-13e)

- **日付**: 2026-07-11
- **状態**: 採用(依頼者要望「LINE のようにページの内容を表示したい」)

### 背景

チャット本文の URL をクリック可能にし、LINE のようにページの内容
(タイトル・説明・画像)をリンクの下に出したい。LINE 等はサーバーが
ページ情報を取得して配るが、PeerCove にはサーバーが無い。

### 選択肢

1. **表示する端末が自分で取得する(OGP メタデータの直接取得)**
2. 送信側が取得してチャットに埋め込んで配る(プロトコル変更が要る。
   送信側が偽のプレビューを配れる)
3. ホストが代理取得して配る(ホストに URL 閲覧が集中・ホスト負荷)

### 決定

**選択肢 1**。表示側の UI(Tauri の Rust 側)が URL へ HTTP(S) アクセスし、
OGP(og:title / og:description / og:site_name / og:image)と `<title>` を
読む。プロトコル・デーモンは変更しない。

- 実装: `link_preview` コマンド(reqwest + rustls。OpenSSL 非依存)。
  HTML は先頭 512 KiB、画像は 2 MiB まで。画像は CSP で外部直読みを
  許していないため Rust 側で取得して data URI にして返す
- HTML パーサ crate は入れず、meta タグだけの最小手書きパーサ
  (`linkmeta.rs`、ユニットテスト付き)
- **取得ガード**: チャットに URL を書くだけで相手の端末がアクセスしに
  行くため、ループバック・プライベート・リンクローカル・CGNAT の
  IP リテラル、`localhost`、`*.internal` / `*.local` は取得しない
  (トンネル内サービスへの意図しないアクセス防止)。リダイレクト先にも
  同じ検査をかける(最大 5 回)
- **プライバシー**: URL を含むメッセージを表示すると、閲覧者の IP が
  相手サイトに伝わる。アプリ設定(⚙)でオフにできる(既定オン)。
  結果は URL ごとにメモリキャッシュ(取得は 1 回)
- URL のクリックは `open_link` コマンド(tauri-plugin-opener)で既定
  ブラウザに渡す。http/https 以外のスキームは開かない

### 残す限界(README に明記しない細部)

- ドメイン名が内部アドレスへ解決される場合(DNS リバインディング等)は
  防げない。VPN メンバーは招待制の身内である前提の割り切り
- 文字コードは UTF-8 のみ(それ以外は文字化けした項目が出ないだけ)

## ADR-0018: アクセス制御 ACL(ホスト側遮断 + 台帳フィルタ)(M3-10)

- **日付**: 2026-07-11
- **状態**: 採用

### 背景

メンバー間通信の許可/禁止をホストが制御したい(M3-10)。Phase 1 の
ハブ&スポークではメンバー間トラフィックは必ずホストを通るが、
直接通信(ADR-0013)確立後はホストを経由しない。遮断は
**リレー経路と直接経路の両方**に効かなければならない。

### 選択肢

1. **ホスト側で遮断 + 台帳のフィルタ配布**
   - リレー: ホストがメンバー間パケットを転送段階で破棄
     (Windows = デバイス内リレーでピア組を検査 / Linux = iptables FORWARD DROP)
   - 直接経路: ホストが台帳を**メンバーごとにフィルタ**し、遮断相手の
     エンドポイントを配布しない(+ `blocked` フラグ)。メンバーの
     DirectManager はエンドポイントが来ない相手に直接ピアを張らず、
     既存の直接ピアも台帳から外れた時点で解除する(既存の仕組みのまま)
2. メンバー側 AllowedIPs から遮断相手を除外する
   (ホストピアが /24 を持つハブ&スポークでは細分化が必要になり複雑。
   メンバー側の実装を信用する前提にもなる)
3. WG 鍵レベルの遮断(メンバー間は同一トンネル内の IP 通信なので不可能)

### 決定

**選択肢 1**。正本はホスト設定 `host.toml` の `[acl]` セクション:

```toml
[acl]
deny = [["10.100.42.2", "10.100.42.3"]]   # 仮想 IP の組(順不同)
```

- **既定はすべて許可**(deny に載せた組だけ遮断)
- **ホスト⇔メンバーは遮断できない**(コントロールチャネル・台帳配布が
  壊れるため validate で拒否)。組の IP はサブネット内であること
- 検証エラーは設定読込ごと失敗させる(黙って無視しない)。ただし
  **既に存在しないメンバーの IP を含む組は許容**(効果がないだけ)。
  remove-peer / UI の削除はその IP を含む組も掃除する
- 適用は supervisor の 5 秒周期の設定再読込に乗る(ADR-0002 と同じ)

### 実装

- `WgBackend::sync_acl(denied)` を追加(冪等、空で全解除)。
  denied の各要素は「両側の仮想 IP + 広告サブネット(ADR-0014)」
  - **Windows**: デバイスが遮断組(仮想 IP 対)を保持し、デバイス内
    リレー(ADR-0003)で「どのピアから来て、どのピアへ出るか」の組で
    判定して破棄。パケットの送信元 IP でなく**ピアの身元**で判定する
    ため、広告サブネット由来のトラフィックも自然に遮断される
  - **Linux**: `iptables -I FORWARD -i <wg_if> -o <wg_if> -s .. -d .. -j DROP`
    を両方向に設定(subnets を含む組合せ)。down・組の解除で対削除
- 台帳: `LedgerEntry` に `blocked: bool` を追加(既定 false、false なら
  ワイヤに現れない = 旧バージョン互換)。配布(control.rs)は接続中の
  メンバー仮想 IP ごとに「遮断相手のエントリの endpoint を落とし
  blocked を立てる」フィルタをかけて送る
- DirectManager は blocked エントリを desired から除外(防御的。
  endpoint が来ない時点で張らないが、二重に守る)
- UI: ホストはメンバータブの「通信制御」ダイアログで組ごとに許可/遮断を
  切替(peercove-ops で `[acl]` を編集 → 5 秒で反映)。メンバーは
  遮断相手に「通信不可」バッジを表示し、チャット送信・ファイル送信の
  宛先から外す(強制はホスト側。UI は案内)

### 残す限界

- 遮断された 2 者が結託して VPN 外で直接つながることは防げない
  (それは VPN の外の話。ACL は「このネットワーク越しの通信」の制御)
- チャット・ファイル送信も IP 層で遮断されるため、遮断相手宛の送信は
  「失敗」として見える(グループチャットでは遮断相手にだけ届かない)

## ADR-0019: 直接接続試行の無害化と固定間隔再試行(ADR-0013 改訂)

- **日付**: 2026-07-12
- **ステータス**: 実装(実機検証フィードバック起点)

### 背景

M3-10 の実機検証で「いつの間にか直接通信が失敗し続け、試行中はチャット等の
通信も捨てられる」という報告を受けた。原因は ADR-0013 の設計に内在していた
2 点の組み合わせ:

1. **試行中のブラックホール**: 直接ピアを追加した瞬間から相手宛 `/32` の
   AllowedIPs が未確立の直接ピアに向くため、ハンドシェイクが完了しない場合は
   タイムアウト(30 秒)までそのメンバー宛の全トラフィック(チャット・
   ファイル送信・ping)が破棄される。ADR-0013 は「一時的なブラックホールに
   なり得るが 30 秒で収束する」と許容していたが、試行が**失敗し続ける環境**では
   再試行のたびに 30 秒の断が繰り返され、実用に耐えない
2. **再試行タイミングの恒久的なずれ**: パンチング(および同一 LAN でも
   相手 OS のステートフルファイアウォールのピンホール開け)には**両側の試行窓が
   重なる**ことが必要。ADR-0013 の指数バックオフ(5 分 → 10 分 → … → 1 時間)は
   両側のタイマーが独立なため、片側だけ失敗・再起動するなどで一度位相がずれると
   30 秒の窓が二度と重ならず、**回復不能**になる。M3-10 更新でデーモンを
   マシンごとに別々のタイミングで入れ替えたことが引き金になったとみられる

ADR-0013 は「双方の追加タイミングのずれが実際に問題になった場合のみ
明示的調停(選択肢 2)を追加する」としていた。本 ADR はその判断点に当たる。

### 選択肢

1. **試行の無害化 + 固定間隔再試行(調停なし)**: 試行中は AllowedIPs を
   空にしてピアを張り(経路を奪わず、ハンドシェイクだけ走らせる)、確立を
   観測してから `/32` を付与する。失敗しても失うものが無くなるので、
   指数バックオフをやめて**固定間隔**で再試行する。試行窓 45 秒・周期 60 秒に
   すると、両側の位相がどうずれても窓の重なりが最低 2×45−60 = **30 秒**保証され、
   ハンドシェイク再送(5 秒間隔)が必ず双方向で交差する
2. **明示的調停(PunchStart メッセージ)**: ホストが開始タイミングを合わせる。
   確実だがコントロールチャネルに状態機械が増え、ホスト再起動・メッセージ
   ロストの考慮も要る
3. ブラックホールだけ直して(1 の前半)バックオフは維持: 断は消えるが
   「直接化がいつまでも回復しない」問題が残る

### 決定

**選択肢 1 を採用**。ADR-0013 の B-2/B-3 を次のように改訂する:

- **二段階 AllowedIPs**: 直接ピアは `allowed_ips = []` で追加し(プローブ)、
  最終ハンドシェイクの観測後に `/32` を付与して初めて経路を切り替える。
  試行中も中継(ホスト `/24`)が生き続けるため、**試行による通信断はゼロ**になる
  - 確立の反映はメンバーの supervise 周期(5 秒)ごとの判定なので、相手側が
    先に `/32` へ切り替えてから自分が追随するまで最大 1 周期、直接ピア経由の
    受信が AllowedIPs 検査で落ちる窓が残る(TCP の再送で吸収される規模)
- **固定間隔再試行**: 試行タイムアウト 45 秒、再試行間隔 60 秒(失敗しても
  伸ばさない)。窓の重なり保証により、経路が物理的に開通し得る環境なら
  両側の位相に関係なくいずれ確立する。開通し得ない環境(シンメトリック NAT
  同士など)では 60 秒ごとにハンドシェイク再送数パケットを送り続けるだけで、
  帯域・CPU への影響は無視できる
- **バックエンドの `add_peer` を upsert に**: 既存ピアに対しては AllowedIPs の
  更新として働く(Linux はカーネルのピア更新、Windows は自作デバイスの
  ピア表を書き換え。**セッションと roaming 学習済みエンドポイントは維持**)。
  確立時の呼び出しは endpoint を渡さない(学習済みの実エンドポイントを
  台帳の値で上書きしないため)。鍵・PSK の変更は従来どおり remove → add
- **ログの静穏化**: 初回の試行・失敗は INFO、2 回目以降の同一エンドポイントへの
  再試行は DEBUG。経路の状態変化(確立・途絶・解除)は従来どおり INFO。
  UI の経路バッジも再試行中は「確立中…」を出さず「中継」のまま
  (背後で静かに試し続け、確立したら「直接」に変わる)
- あわせて、台帳の再配布(エンドポイント鮮度の 60 秒粒度更新で最大毎分
  数回起きる)をメンバーが受信するたびに INFO を出していたのを、
  **意味のある変化(メンバーの増減・名前・オンライン・遮断・サブネット・
  DNS レコード)があったときだけ INFO**、それ以外は DEBUG に落とす

### 理由

- 調停メッセージなしという ADR-0013 の設計原則を守ったまま、ずれ問題を
  「窓を重ねる」ことで数学的に解決できる(固定周期 P=60、窓 L=45 なら
  任意の位相差で重なり ≥ 2L−P = 30 秒)
- 試行が無害になったことで「積極的に試し続ける」ことが許容され、
  バックオフ(失敗コストの抑制が目的だった)の存在理由自体が消える
- 失敗継続時の唯一のコストはハンドシェイク再送(148 バイト × 5 秒間隔 ×
  45 秒/分)で、keepalive と同オーダー

### 残す限界

- 確立直後の最大 1 周期(5 秒)、相手からの直接受信が落ちる窓がある(上述)
- シンメトリック NAT 同士は従来どおり直接化できない(60 秒周期の試行が
  静かに続く。無駄を止めたい場合は `direct = false` で明示的に切る)

## ADR-0020: メンバー鍵ローテーション(M3-11)

- **日付**: 2026-07-12
- **状態**: 承認済み(2026-07-12 依頼者承認。自動 1 回 + 手動、ホスト鍵・PSK・定期は先送り)

### 背景

参加方式(ADR-0005 案 B)ではメンバーのデバイス秘密鍵を**ホストが生成し、
招待トークンに同梱して**渡している。つまり:

1. メンバーの秘密鍵が**トークンという文字列でチャット・QR などの経路を通る**
   (経路上に残る可能性がある)
2. **ホストがメンバーの秘密鍵を知っている**(直接通信 ADR-0013 の導入後は
   メンバー間トラフィックがホストを経由しないため、「どうせ全部ホスト経由」
   という当初の免罪符が弱くなっている)

ADR-0005 はこれを「鍵ローテーションは将来コントロールチャネル上で実装可能」
として先送りしていた。M3-11 はその回収で、**メンバーが自分のデバイス上で
新しい鍵ペアを生成し、秘密鍵を一切外に出さずに公開鍵だけをホストへ届けて
差し替える**ことをゴールとする。

前提となる既存の仕組み:

- ホストの supervisor は 5 秒ごとに host.toml を再読込し、`[[peer]]` の
  フィンガープリント変化を「削除 + 再追加」で実行中バックエンドに反映する(M1-7)
- コントロールチャネルは追加メッセージに対して前方互換(未知メッセージは
  黙って無視、PROTO_VERSION は上げない)
- 台帳の公開鍵が変われば、他メンバーの直接通信(direct.rs)は「旧キー消滅 →
  新キー出現」として自然に追随する

### 選択肢

**トリガー(いつ回すか)**

1. **参加直後に自動で 1 回**(+ 手動での再実行)… トークンを経由した鍵を
   最初の接続で自動的に退役させる。ユーザーは何も意識しない
2. 手動のみ(UI ボタン / CLI)… 自動の挙動変化がない代わりに、大半の
   ユーザーは一度も回さず、トークン露出の問題が残り続ける
3. 定期(例: 30 日ごと)… WG の静的鍵は Noise により前方秘匿性が
   セッション単位で確保されるため、定期入れ替えの追加価値は小さい

**新公開鍵の永続化(ホスト側)**

- A. **host.toml の `[[peer]].public_key` を書き換える**(正本を維持)。
  デーモンが初めて設定ファイルを書くことになるため、所有権・権限を保った
  上書き(in-place write)と、UI(ops)の編集との希少な競合を受け入れる
- B. デーモン専用のサイドカーファイルに「鍵の上書き表」を持つ … host.toml に
  古い公開鍵が残り続け、台帳導出・remove-peer・invite の突合が二重帳簿になる

### 決定(提案)

**スコープ: メンバーのデバイス鍵(X25519)のみ。トリガーは「参加直後に自動 1 回
+ 手動」。永続化は案 A(host.toml が正本のまま)。**

1. **プロトコル**(追加メッセージ、PROTO_VERSION は上げない):
   - member → host: `rotate_key { new_public_key }`
   - host → member: `rotate_key_result { accepted, message }`
   - ホストは接続元仮想 IP でメンバーを特定(既存の身元モデル)。新公開鍵が
     他ピア・ホストと衝突すれば拒否。**現在の登録鍵と同じなら成功扱い(冪等)**
2. **ホスト側の適用順序**: host.toml へ永続化 → `rotate_key_result` 送信 →
   supervisor の次回再読込(≤5 秒)が WG ピアを削除 + 再追加。結果送信が
   先なので、旧鍵のセッションが生きているうちに応答が届く。PSK・name・IP・
   subnets・ACL は変更しない(PSK は静的鍵と独立なので据え置きでよい)。
   ピア同期(M1-7)は「消えた鍵の IP を別の鍵が引き継いだ」を**鍵の
   入れ替え**として扱う: 削除通知(Removed)を送らず(送るとメンバーが
   「削除された」と誤認して再接続をやめる)、同じ /32 のピアを並存させない
   よう旧鍵の除去を新鍵の追加より先に行う
3. **メンバー側の適用と障害回復**(すべてデーモン内で完結):
   - 新鍵を生成し **`member.key.new` に書いてから** `rotate_key` を送る
     (クラッシュしても秘密鍵を失わない)
   - `accepted` を受けたら member.key を新鍵で上書き → member.key.new を
     削除 → トンネルを内部で down/up(数秒の断)
   - 応答が来ない(旧ホスト or 直後の切断)場合は member.key.new を**残して**
     通常運転を続ける。以後は「**ホストからの受信が 45 秒止まったら**もう一方の
     鍵で入れ直す(交互試行)」で、どちらの鍵をホストが持っていても自力で
     収束する(判定に最終ハンドシェイク経過を使わないのは、通常運転でも
     2 分まで伸びるため。健全なら keepalive 25 秒への受動 keepalive で
     35 秒以内に必ず受信がある)。現行鍵で繋がった場合は同じ
     new_public_key で `rotate_key` を再送(冪等なので安全)
4. **「参加直後に自動 1 回」の判定**: member.toml の `key_source` が
   `"self"` でなければ自動ローテーションし、完了時にデーモンが
   `key_source = "self"` を書く。**フィールドがない設定は "token" 扱い**
   (既存メンバーの鍵もすべてトークン経由なので、アップデート後の初回接続で
   一度だけ自動更新される)。join はこのフィールドを**書かない**
   (書くと参加直後の member.toml が旧デーモンで読めなくなるだけで、
   省略時の既定と同じ意味のため)
5. **手動トリガー**: IPC に `rotate_key { config }` を追加(応答は Done)。
   UI はメンバーとして参加しているネットワークの詳細に「鍵を更新」操作を置く
6. **デーモンによる設定ファイル書き込みの原則**(新設): 対象は
   member.key / member.toml の `key_source` / host.toml の `public_key` のみ。
   既存ファイルは所有権・権限を保つ in-place 上書き(Linux で root が
   ユーザー所有ファイルを書き換えても所有者を変えない)。新規作成する
   member.key.new は member.key の所有者・権限に合わせる

### 理由

- 自動 1 回(選択肢 1)にするのは、この機能の価値が「トークン経由の鍵の退役」
  そのものだから。手動のみでは能動的に回す人にしか効果がない
- 案 A(host.toml 正本)は台帳導出・remove-peer・ピア同期(M1-7)・invite の
  突合をすべて既存コードのまま使える。デーモンと UI の書き込み競合は
  「メンバーの初回接続時 or 手動時に 1 回」しか起きないので実害が小さい
- 応答→適用の順序と「pending 鍵を消さない + 両鍵フォールバック」の組で、
  どのタイミングでどちらが落ちても鍵を失って締め出される状態がない
  (最悪でも T 秒 + 再接続で収束)
- 旧ホスト相手では `rotate_key` が無視され、タイムアウト後も現行鍵のまま
  動き続ける(互換性を壊さない)

### 対象外(将来課題)

- **ホスト鍵のローテーション**: 全メンバーの member.toml 更新が必要で、
  ローテーション時にオフラインだったメンバーが恒久的に締め出される
  (新旧鍵の併走期間の設計が要る)。必要になった時点で別 ADR
- **PSK のローテーション・メンバー間 PSK**(ADR-0013 の再検討事項):
  静的鍵の入れ替えとは独立なので、耐量子性の要件が出た時点で検討
- **定期ローテーション**: 上記「選択肢」の 3 のとおり追加価値が小さい
- UI のアバター色は公開鍵から導出しているため、ローテーション後に変わる
  (実害はないが、気になるようなら IP 由来に変える余地あり)

## ADR-0021: 表示名と DNS 名の分離・固定化(M3-14a)

- **日付**: 2026-07-12
- **状態**: 採用(依頼者要望 2026-07-12。方針承認済み: DNS 名は本人が変更可 +
  ホストが全員分を管理、レコード類はホスト管理のみ)

### 背景

現行(ADR-0011)は DNS 名を**表示名から毎回導出**する(正規化 → 空なら
`host`/`member-<第4オクテット>` → 衝突は IP 順で自動リネーム)。依頼者要望:

1. 表示名(日本語・空白可)と DNS 名(`a-z0-9-`)を**別設定**にし、個別に変更できる
2. DNS 名は仮想 IP が変わっても維持される(固定化)
3. 重複は自動リネームではなく**設定時エラー**、予約語(`localhost` 等)は拒否
4. 初期値は表示名・DNS 名とも `host` / `member-2` / `member-3` など

### 選択肢

**DNS 名の正本の置き場所**

- A. **host.toml を正本にする(採用)** … `[interface].dns_name`(ホスト自身)+
  `[[peer]].dns_name`(メンバー)。台帳(`LedgerEntry.dns_name`)で全員へ配布。
  表示名(`[[peer]].name`)・subnets・ACL と同じ「ホスト正本 → 台帳配布」パターン
- B. 各メンバーの member.toml に置き Hello で申告 … 重複検証の正本が分散し、
  衝突時にどちらが勝つかが接続順依存(非決定的)になる
- C. 専用ファイル(dns.toml) … 正本が 2 か所に割れ、invite/remove との突合が二重帳簿

**メンバー本人による変更の経路**

- A. **コントロールチャネルに追加メッセージ(採用)** … `set_dns_name { name }` →
  ホストが検証・永続化 → `set_dns_name_result { accepted, message }`。
  ADR-0020 の `rotate_key` と同じ「メンバー発 → ホストが host.toml へ適用」パターン
- B. ホスト UI のみで変更 … 承認済み方針(本人が変更可)を満たさない

### 決定

1. **設定**: `InterfaceConfig.dns_name: Option<String>`(ホスト自身用)と
   `PeerConfig.dns_name: Option<String>` を追加。`deny_unknown_fields` のため
   これを書いた host.toml は旧バージョンでは読めない(subnets/ACL と同じ扱い)
2. **台帳**: `LedgerEntry.dns_name: Option<String>` を追加。互換規則は endpoint と
   同じ(None ならワイヤに現れない・旧バージョンとは互いに無視し合う。
   PROTO_VERSION は上げない)
3. **ゾーン導出**(`zone_for`): エントリに `dns_name` があれば**そのままラベルに
   採用**(ホスト検証済み。防御的な一意化は維持)。無ければ**従来導出に
   フォールバック**(旧ホストの台帳・未設定エントリで名前が変わらない)
4. **検証**(core `names.rs` に一元化): 入力を正規化(小文字化・空白/記号 →
   ハイフン)→ 空になれば「英数字を含めてください」エラー → 予約語
   (`localhost` `local` `internal` `arpa` `peercove` `dns` `gateway` `relay`、
   および `host` はホスト専用)→ ホストが host.toml 全体(interface + 全 peer +
   カスタムレコード)と突合して重複はエラー。カスタムレコード追加(ops::dns)にも
   同じ予約語検証を適用する
5. **初期値と固定化**: invite が IP 割当と同時に DNS 名を確定・永続化する
   (表示名の正規化ラベル、空なら `member-<第4オクテット>`、重複時は従来と同じ
   `-<oct>` 一意化)。以後この値は IP と独立に維持される。**既存(アップグレード
   前に登録された)ピアは dns_name 無し = 従来導出のまま**とし、一括移行は
   しない(現行名が変わらない。編集した時点で確定・永続化される)
6. **変更経路**:
   - ホスト: ops に `set_host_dns_name` / `set_peer_dns_name`(toml_edit で
     in-place 更新、rename_peer と同型)。UI のメンバー管理・設定画面から呼ぶ。
     反映は supervisor の次回再読込(≤5 秒)で台帳に載る(再起動不要)
   - メンバー本人: UI → IPC `set_dns_name { config, name }` → デーモンが
     コントロールチャネルで `set_dns_name` 送信 → ホストが検証・永続化して
     `set_dns_name_result` を返す(接続中のみ変更可。旧ホスト相手は無視される
     ためタイムアウトで「ホストが未対応」と案内)
7. **表示名は現行のまま**(host.toml `[[peer]].name` 正本、日本語可)。
   本 ADR 以降、表示名の変更は DNS 名に**影響しない**(dns_name 確定済みの
   エントリでは)

### 理由

- 案 A は「ホスト正本 → 台帳配布」という既存の一貫パターンに乗り、
  重複検証の正本が 1 か所(host.toml)に定まる
- ゾーン導出のフォールバック維持により、新旧バージョン混在でも解決が壊れない
  (旧ホスト × 新メンバー = 全員従来導出 / 新ホスト × 旧メンバー = 旧メンバーは
  dns_name フィールドを無視して従来導出 — 同じ台帳から同じ名前になるよう、
  新ホストは dns_name をゾーン導出の入力にも使うが従来導出結果と大きく
  乖離しない初期値を与える)
- invite 時確定(遅延割当でなく)により「IP が変わっても名前は変わらない」を
  エントリの生存期間全体で保証できる

### 対象外(将来課題 = M3-14b 以降)

- エイリアス・サービス名・端末配下サブドメイン・LAN 機器レコード(14b/14c)
- 逆引き(14d)、オンライン連動・ヘルスチェック(14e)、ワイルドカード(14f)、
  短縮名 = 検索サフィックス(14g)

## ADR-0022: DNS レコードモデル拡張(M3-14b)

- **日付**: 2026-07-12
- **状態**: 採用(依頼者要望 10/11/12/14。方針承認済み: レコード類はホスト管理のみ)

### 背景

カスタムレコード(ADR-0011 §1b)は `{name(単一ラベル), ip(固定)}` のみで、
依頼者要望の以下を表せない:

- **10 エイリアス**: 1 台に別名を複数付けたい(`gamehost` → 山田さんの PC)
- **11 サービス名**: 実行端末が変わっても同じ名前で届いてほしい(参照先の付け替え)
- **12 端末配下サブドメイン**: `web.host.<net>.peercove.internal` のような階層
- **14 LAN 機器**: サブネットルーター(ADR-0014)配下の機器に名前を付けたい。
  広告範囲内の IP のみ登録可

固定 IP しか持てないため、参照先メンバーの仮想 IP が変わる(削除 → 再招待)と
レコードが黙って古くなるのが最大の欠陥。

### 選択肢

**メンバー参照(IP 追随)の実現方法**

- A. **ホストが配布時に IP へ解決する(採用)** … 設定(host.toml)にはメンバー参照を
  持ち、台帳を配布する 5 秒周期でその時点の仮想 IP・DNS ラベルに解決してから
  従来のワイヤ形式 `{name, ip}` で配る。**ワイヤ変更ゼロ**で、旧メンバーも
  エイリアスをそのまま解決できる
- B. ワイヤの `DnsRecord` にメンバー参照を足し、各メンバーが解決 … 解決ロジックが
  全ノードに複製され、旧メンバーは新形式レコードを一切解決できない

**メンバー参照の識別子**

- A. **公開鍵(採用)+ 鍵ローテーション時に参照を書き換え** … `[[peer]]` の主キーと
  同じ。DNS 名の改名では壊れない。鍵ローテーション(ADR-0020)で変わるため、
  `rotate_peer_key` が同じ書き込みで `[[dns_record]]` 内の参照も差し替える
- B. DNS 名 … 人間可読だが、未確定(アップグレード前)のメンバーを参照できず、
  改名時に参照の書き換えが要る(改名は鍵ローテーションより高頻度)
- C. 仮想 IP … 削除 → 再招待で別人に付け替わる(まさに防ぎたい事故)

ホスト自身への参照は公開鍵が設定に無いため、番兵文字列 **`"host"`** を使う。

### 決定

1. **設定**: `[[dns_record]]` を拡張する(型は `DnsRecordConfig`)。
   ```toml
   [[dns_record]]
   name = "gamehost"          # 正規化済みラベル(必須)
   ip = "192.168.1.50"        # ターゲット A: 固定 IP(従来型)
   member = "host" | "<公開鍵>" # ターゲット B: メンバー参照(IP 自動追随)
   under = "host" | "<公開鍵>"  # 任意: 親メンバー(端末配下サブドメイン)
   ```
   `ip` / `member` は排他でどちらか必須。旧設定(`name` + `ip`)はそのまま読める。
   `member` / `under` を書いた設定は旧バージョンでは読めない
   (`deny_unknown_fields`。subnets / ACL / dns_name と同じ扱い)
2. **用途との対応**: エイリアス(10)= `member` ターゲットの最上位レコード。
   複数別名は複数レコードで表す(専用のエイリアス型は持たない)/
   サービス名(11)= 同(移設は参照先の付け替え。ポート等のサービス情報は 14c)/
   端末配下サブドメイン(12)= `under` 付き `member` ターゲット
   (`name = "web", under = <alice>` → `web.<aliceのラベル>.<net>.peercove.internal`)/
   LAN 機器(14)= `under` 付き `ip` ターゲット。**ip はその親メンバーの広告
   サブネット内のみ許可**(要望 14 の検証)。ホストは広告サブネットを持たない
   ため `under = "host"` + `ip` は不可 — ホスト側 LAN 機器は従来どおり最上位の
   `ip` レコード(範囲制限なし = 既存互換)
3. **配布(ワイヤ変更なし)**: ホストが台帳構築と同時にレコードを解決する
   (`member` → その時点の仮想 IP、`under` → その時点の DNS ラベル)。配布形式は
   従来の `{name, ip}` のままで、サブドメインは `name` がドット付き相対名
   (`web.alice`)になるだけ。参照先が見つからないレコードは配布から外す(防御)。
   旧メンバーはドット付き名を `zone_for` の検証で黙って飛ばす
   (そのレコードだけ解決不可 = 無害な劣化)。新メンバーの `zone_for` は
   ドット区切りの各ラベルが正規形なら受け入れる(`is_dns_name`)
4. **一意性と予約語**: レコードは **(name, under) の組で一意**(親が違えば同名可、
   `web.host` と `web.alice` は共存)。最上位(under なし)の name は従来どおり
   メンバーの DNS ラベル・予約語との重複を拒否。under 付きの name に予約語
   チェックは適用しない(予約は最上位の紛らわしさが対象)
5. **参照の追随**: `rotate_peer_key`(鍵ローテーション)は `[[dns_record]]` 内の
   `member` / `under` も同じ書き込みで新しい公開鍵へ差し替える。`remove_peer` は
   削除メンバーを参照するレコードを掃除する(ACL の掃除と同様)
6. **UI**: DNS 管理画面の追加フォームを拡張(ターゲット = IP 直指定 / メンバー選択、
   配置 = 最上位 / メンバー配下)。一覧は解決後の FQDN と参照先(メンバー名 or IP)
   を表示。削除は (name, under) で特定する
7. **CLI は追加しない**(現状レコード管理は UI のみ。必要になったら別途)

### 理由

- ホスト解決方式(選択肢 A)は「ホスト正本 → 台帳配布」の既存パターンそのままで、
  ワイヤ・IPC・旧バージョンへの影響がゼロ。台帳は 5 秒周期で差分再配布される
  仕組みが既にあるため、「IP 変更に追随する」が追加コストなしで手に入る
- 公開鍵参照は host.toml 内で完結して検証でき(`Config::validate` で存在確認)、
  DNS 名未確定の旧メンバーも参照できる。鍵ローテーションは同一ファイルへの
  1 回の書き込みで整合を保てる
- (name, under) 一意化により、要望 12 の「端末ごとに web を持つ」が自然に書ける

### 対象外(将来課題)

- サービス情報(ポート・プロトコル・URL コピー UI)= 14c
- 逆引き(14d)、オンライン連動・ヘルスチェック(14e)、ワイルドカード(14f)、
  短縮名 = 検索サフィックス(14g)
- レコードの多段ネスト(`a.b.<member>`)。1 レコード = ラベル 1 つ + 親 1 段のみ

## ADR-0023: DNS レコードのサービス情報と URL コピー(M3-14c)

- **日付**: 2026-07-12
- **状態**: 採用(実装済み、実機検証待ち)

### 背景

ADR-0022 でカスタム DNS レコードはエイリアス・サービス名・端末配下
サブドメイン・LAN 機器を表せるようになったが、配布される情報は名前と IP
だけだった。利用者はサービスへ接続するときにスキームとポートを別途覚え、
`http://gamehost.home.peercove.internal:8080/` のような URL を手入力する必要が
ある。依頼者要望 13 とサービス名要望 11 の残りとして、DNS 管理画面から
接続先 URL をそのままコピーできるようにする。

### 選択肢

1. **A レコードにスキーム/ポートのメタ情報を添えて台帳配布する(採用)**。
   名前解決は従来の A レコードのまま、PeerCove UI だけがメタ情報から URL を
   組み立てる
2. **DNS の SRV レコードとしてサービス情報を配信する**。DNS 標準の表現だが、
   一般的なブラウザや OS リゾルバは Web URL を開く際に SRV を参照しない。
   既存の最小 DNS サーバーにも SRV 問い合わせ・応答の実装が必要になる
3. URL 文字列を設定・台帳へそのまま保存する。組み立ては不要だが、DNS 名を
   改名したときやメンバー配下の親ラベルが変わったときに URL が古くなり、
   名前と URL が二重帳簿になる

### 決定

1. **設定**: `DnsRecordConfig` に任意の `scheme: Option<String>` と
   `port: Option<u16>` を追加する。スキームは先頭が小文字英字、以降が
   小文字英数字または `+.-`、31 文字以内。ポートは 1〜65535。
   ops での追加時はスキームを小文字化してから検証・保存する。
   `deny_unknown_fields` のため、これらを書いた host.toml は旧バージョンでは
   読めない(他の新設定フィールドと同じく明示エラーになる)
2. **ワイヤ**: 配布用 `DnsRecord` に同じ任意フィールドを追加し、
   `None` はシリアライズしない。旧メンバーは未知フィールドを無視して従来どおり
   A レコードを解決し、新機能の URL 表示だけが使えない。`PROTO_VERSION` /
   `IPC_VERSION` は上げない。サービス情報なしのレコードは旧 JSON と完全に一致する
3. **配布**: ホストの `resolve_records` はメンバー参照を IP/相対名へ解決するとき、
   `scheme` / `port` を変更せずコピーする。DNS サーバーと `zone_for` は
   引き続き名前/IP だけを使い、DNS 応答形式は A レコードのみとする
4. **URL 組み立て**: core の純関数に一元化する。スキームがあれば
   `{scheme}://{fqdn}{:port?}/`。`http:80` と `https:443` はポートを省略する。
   スキームなし + ポートありは URL にせず、UI が `fqdn:port` と表示する。
   どちらも無ければ従来表示のまま
5. **UI**: ホストの追加フォームに自由入力可能なスキーム(`http` / `https` を
   候補表示)とポートを追加する。ホスト・メンバー共通の一覧に組み立て済み URL と
   「URL をコピー」を表示する。メンバー側は引き続き閲覧のみ

### 理由

- 選択肢 1 は旧バージョンの名前解決を維持したまま UI の利便性だけを追加できる。
  `DnsRecord` は未知フィールドを許容し、任意フィールドを省略する既存の
  前方互換パターンにそのまま乗る
- SRV はサービス発見には適するが、ブラウザ・一般的な OS リゾルバが URL の
  ポート決定に利用しないため「ワンクリックで URL コピー」という目的を満たさない。
  DNS サーバーを A レコード専用のまま保つ方が実装・検証範囲も小さい
- URL を導出値にすることで、ADR-0022 の IP 追随や親 DNS 名変更に自然に追随し、
  設定内の二重帳簿を避けられる

### 対象外

- SRV 等の DNS プロトコル拡張、CLI のレコード操作コマンド
- 逆引き(14d)、オンライン連動・ヘルスチェック(14e)、短縮名 = 検索サフィックス(14g)
- ワイルドカード(14f)は当初対象外だったが、検証 FB で前倒し実装 → ADR-0024

## ADR-0024: DNS レコード UI の再構成とワイルドカード(M3-16)

- **日付**: 2026-07-13
- **背景**: 検証 FB。ADR-0022 の DNS 追加 UI は「ターゲット(IP/メンバー)」と
  「配置(最上位/メンバー配下)」が別軸で分かりにくかった。要望は
  (a) ドメイン欄を `<net>.peercove.internal` 固定 + その左を自由に、
  (b) 転送先は「マシン or IP」で選ぶ、(c) ワイルドカード DNS 対応、
  (d) 端末の DNS 名変更時もカスタムドメインとの重複を判定。

### 決定

1. **ドメイン名の構成**(左→右): `<prefix>.<base>.<net>.peercove.internal`
   - `base`(net 直下): **自由入力**(任意ラベル)/ **マシン選択**(そのマシンの
     DNS ラベルに解決 = サブドメイン。マシン改名にも追随)のどちらか。
   - `prefix`(さらに左): 空可の自由入力。**先頭ラベルのみ `*`** でワイルドカード、
     ドットでサブドメインも書ける。
   - 内部表現は ADR-0022 の (name, under) を流用: base=マシンなら
     `under=MemberRef`・`name=prefix`、base=自由入力なら `under=None`・
     `name=prefix.base`(結合)。**ワイヤ・IPC・設定スキーマは無改訂**。
2. **転送先**(この名前が指す IP): **マシン参照**(その時点の IP に追随)/
   **IP 直指定** の排他。ADR-0022 の ip/member をそのまま使う。
3. **ワイルドカード解決**: 内蔵リゾルバは完全一致を優先し、無ければ**先頭 1 ラベルを
   `*` に置換**して 1 段だけ再探索(`foo.app.<net>` → `*.app.<net>`)。ゾーン導出は
   `is_custom_dns_name`(先頭 `*` を許すラベル検証)で受け入れる。多段
   (`a.b.app`)は非対応。
4. **予約語・重複**: net 直下のラベル(= 相対名の末尾ラベル)が `host` /
   予約語 / メンバーの DNS ラベルと衝突するものは追加を拒否。逆に端末の DNS 名
   変更は従来どおり `taken_dns_labels`(最上位カスタム名を含む)で弾く。
5. **配置(under)は UI から廃止**。旧レコード(under 付き・LAN 機器)は
   引き続き読める(後方互換)。

### 理由

- (name, under, ip/member) の既存モデルに載せることで、ワイヤ・設定の互換を保ったまま
  UI だけを分かりやすくできる。ワイルドカードは配布形 `{name, ip}` の name に `*` を
  含めるだけで済み、リゾルバの 1 行追加で解決できる。

### 対象外

- 多段ワイルドカード、`*` を途中ラベルに置く記法、ワイルドカードの URL/サービス表示

## ADR-0025: 転送先 CNAME(外部ドメイン対応、M3-17)

- **日付**: 2026-07-14
- **背景**: 検証 FB。転送先に IP・マシンだけでなく**別ドメインの別名(CNAME)**を
  持ちたい。外部ドメイン(`docs` → `example.com`)も対象。

### 決定

1. **モデル**: `DnsRecordConfig` に `cname: Option<String>` を追加。ターゲットは
   `ip` / `member` / `cname` の**ちょうど 1 つ**(config 検証で強制)。CNAME の
   ターゲットは小文字化・末尾ドット除去して各ラベルを検証(外部ドメイン可)。
2. **配布(後方互換)**: A レコードとは**別枠**で配る。`ControlMessage::Ledger` に
   `cname_records`(`#[serde(default, skip_serializing_if = Vec::is_empty)]`)を
   追加。旧メンバーは未知フィールドとして無視 → CNAME を解決しないだけ(壊れない)。
   `PROTO_VERSION` / `IPC_VERSION` は上げない。ホストは `resolve_cnames` で `under`
   を解決してから配る。
3. **フラット化(検証 FB で追加)**: 当初は CNAME RR をそのまま返してスタブ
   リゾルバに追跡させる設計だったが、**Windows は split DNS 越しに外部 CNAME 先を
   たどれない**(先の問い合わせが内蔵サーバへ行き NXDOMAIN)ことが実機で判明。
   そこで**ホストが転送先を IPv4 へ解決**して `CnameRecord.resolved_ip` に載せ、
   台帳と一緒に配る(3 秒タイムアウト・5 分キャッシュ・失敗時は直近値を保持)。
4. **内蔵リゾルバ**: 共有ゾーンを A(`fqdn→IP`)と CNAME(`fqdn→(ドメイン, 解決 IP)`)
   の 2 表に。応答は **A を優先**(RFC 1912)。CNAME は、**解決 IP があれば A で返す
   (フラット化 → 外部でも到達可)**、無ければ CNAME RR(type 5)で返す(in-zone
   先・解決前のフォールバック)。CNAME にも 1 段ワイルドカードを適用。
5. **UI**: 転送先に「ドメイン(CNAME)を指定」を追加。一覧は別名先ドメインを表示。

### 理由

- 別枠配布で旧バージョンとの互換を保ちつつ、A/CNAME 共存不可の原則(A 優先)を
  リゾルバ側で担保。フラット化により、非再帰の最小リゾルバのまま外部ドメインへ
  到達できる(ホストだけが外部 DNS を引き、メンバーは配られた A を返すだけ)。

### 対象外

- CNAME チェーン(多段)、CNAME の TTL 個別指定、AAAA(IPv6)フラット化、MX/TXT 等

## ADR-0026: バイナリ/クレートを peercove-poc から改名(M3-18)

- **日付**: 2026-07-14
- **背景**: `poc` は M0 技術検証(proof of concept)の名残。M0/M1 完了・製品化が進む
  現状で実態と合わない。ADR-0010 では「バイナリ名は `peercove-poc` のまま M2 を出す
  (製品名へのリネームは後回し)」と決めていた(§6)。その保留を解消する。

### 決定

1. **クレート**: `crates/peercove-poc` → `crates/peercove-cli`(`package.name = "peercove-cli"`)。
   `cargo … -p peercove-cli` で参照する。
2. **バイナリ**: `[[bin]] name = "peercove"` を明示。生成物は `peercove(.exe)`。
   コマンドは `peercove daemon run` / `peercove host --config …` と製品名そのまま。
   なお `module_path!()`(= `tracing` の target)は**パッケージ名でなくバイナリ名**に
   従うため `peercove::daemon` 等になる(ログ target と関連テストもこれに合わせる)。
3. **互換に影響なし**: OS/通信レベルの名前は既にクリーンで**変更しない** ——
   Windows サービス名 `peercove-daemon`、名前付きパイプ `\\.\pipe\peercove-daemon`、
   Unix ソケット `/run/peercove.sock`、systemd ユニット `peercove-daemon.service`。
   よってサービス登録・IPC・ワイヤ互換はいずれも壊れない。
4. **追随**: `ExecStart` / Tauri サイドカー / MSI(.wxs)/ deb / ZIP・tar / README・
   CLAUDE.md・AGENTS.md・roadmap を新バイナリ名へ更新。過去のハンドオフ文書と本
   decisions.md の既存 ADR 本文(ADR-0010 §6 等)は当時の記録として原文保持。

### 理由

- 名前を実態(製品 CLI + デーモン)に合わせる。クレート名 `-cli` と製品バイナリ名
  `peercove` を分けることで、`cargo … -p peercove-cli` とユーザー向け `peercove …`
  の役割が明確になる。OS 統合名を据え置くことで、既存インストールの再登録も不要。

### 対象外

- UI 実行ファイル名(`PeerCove` / `peercove-ui`)、サービス名・IPC 名の変更

## ADR-0027: 表示名変更のメンバー一覧一元化とメンバー本人の表示名変更(M3-19)

- **日付**: 2026-07-14
- **状態**: 採用(検証 FB 2026-07-14)
- **背景**: 検証 FB。(1) 表示名・DNS 名の変更が**メンバー一覧と設定画面の 2 か所**に
  あり紛らわしい → メンバー一覧に一元化したい。(2) メンバー一覧に**自分の表示名を
  変更する操作**が無い(ホストは他メンバーを変更できるが、メンバーは自分を変更
  できない)。(3) **Linux メンバーが設定画面で表示名を変えても反映されない**。

### 原因(不具合 3)

表示名は host.toml `[[peer]].name` が正本(ADR-0021 点7)で、台帳は
`build_ledger` が常に `[[peer]].name` から組む。メンバーの member.toml
`[interface].display_name` は Hello の名乗り(ログ 1 行)にしか使われず、
**台帳には一切反映されない**。設定画面の表示名編集は member.toml を書くだけの
**実質 no-op** だった(ログも出ない = 何も起きていない)。

### 決定

1. **変更経路(表示名)**: DNS 名(ADR-0021)と同じ「メンバー発 → ホストが
   host.toml へ適用」に統一する。
   - コントロールチャネルに `set_display_name { name }` / `set_display_name_result`
     を追加(proto。追加メッセージなので `PROTO_VERSION` は上げない。旧ホストは
     解析失敗で無視 → メンバーはタイムアウトで「未対応」と案内)。
   - IPC に `SetDisplayName { config, name }` を追加(`IPC_VERSION` は上げない)。
   - ホストは `set_peer_display_name_by_ip`(= 仮想 IP で特定 → `rename_peer`)で
     `[[peer]].name` を書き、台帳の再配布(≤5 秒)で全員に伝わる。
   - ホスト自身は `set_host_display_name`(`[interface].display_name` を直接)。
2. **UI 一元化**: 表示名・DNS 名の変更は**メンバー一覧の自分の行の ✎** に集約し、
   設定画面(NetworkSettingsView)からは削除。
   - 自分の行の表示名 ✎: ホスト = `set_host_display_name` / メンバー =
     IPC `set_display_name`(接続中のみ)。他メンバーの行(ホストのみ)は従来の
     `rename_member`(host.toml 直接)。
   - 自分の行の DNS 名 ✎ は既存(ADR-0021)のまま。
3. **表示文言**: アクションの ✎ ツールチップを「名前を変更」→「表示名を変更」に。

### 理由

- ADR-0021 が DNS 名で確立した「メンバー発 → ホスト適用」パターンにそのまま乗る。
  正本(host.toml)が 1 か所のまま、本人変更・ホスト変更のどちらも同じフィールドへ
  収束する(最後の書き込みが勝つ)。
- 変更 UI が 1 か所になり、「設定画面で変えたのに反映されない」不整合が消える。

### 対象外

- 表示名の初期割当(invite)経路は不変。member.toml `[interface].display_name` は
  当面は Hello の名乗り用に残す(台帳の正本ではない)。

## ADR-0028: 設定画面に実行時のインターフェース名を表示(M3-20)

- **日付**: 2026-07-14
- **状態**: 採用(検証 FB 2026-07-14)
- **背景**: 同一 Windows ホストで 2 ネットワークをホストすると、設定画面の
  「インターフェース」が両方 `peercove0` と出る。実際は起動時の自動採番
  (ADR-0012 の `plan_interface`)で片方が `peercove1` になっており、OS 上は
  別アダプタで正しく共存している(`ipconfig` でも確認)。表示が設定ファイルの
  `interface.name`(既定 `peercove0`)を出していて実態と食い違う、という UI の
  ズレ(機能影響なし)。

### 決定

1. **実行時名を status に載せる**: `TunnelInfo.interface_name`(`#[serde(default)]`。
   旧デーモンは空 → UI は従来どおり設定値を出す)。デーモンは `Active.if_name`
   (= `plan_interface` の結果)を入れる。DTO `Tunnel.interfaceName` へ通す。
2. **設定画面の表示**: 接続中(`Tunnel` がある)は `interfaceName` を優先表示し、
   設定値と異なるときは「自動採番」の注記を添える。未接続は従来どおり設定値。

### 理由

- 正本(host.toml の `interface.name`)は変えず、表示だけ実態に合わせる最小変更。
  ワイヤ・IPC は追加フィールドのみで互換を維持(バージョンは上げない)。

### 対象外

- 設定画面からのインターフェース名の編集(現状どおり init/設定ファイル直編集)。

## メモ: 仮想 IP レンジと Tailscale の衝突(M1 で要検討)

- **日付**: 2026-07-07(G-2 実機検証で発覚)

仮想 IP の既定例 `100.100.42.0/24`(handoff 2 章)は CGNAT レンジ
`100.64.0.0/10` 内にあり、**Tailscale 導入済みマシンでは同レンジの送信元を持つ
パケットが `ts-input` ルールで DROP される**(WG 復号後・IP スタック配達前)。
G-2 検証では member VM に Tailscale が入っており、handshake・転送量は正常なのに
ping だけ落ちる、という切り分けの難しい症状になった。

M0 では回避策(Tailscale 停止 or 設定でレンジ変更)を README に記載した。
**M1 で既定レンジを再検討すること**。候補: RFC1918 の衝突しにくい帯
(例 `10.64.x.0/24` をランダム生成)や、招待トークンにサブネットを含めて
ホストが決める方式。宅内 LAN(192.168.0.0/16 の一部)や他 VPN との衝突検出も
セットで検討する。
